### Summary

Phase 5 plans aim to implement a true single-process wfb_core architecture, replacing the multi-process wfb_tx/wfb_rx/wfb_tun approach. The plans are comprehensive, covering command-line parsing, integration of RX/TX/TUN threads, scheduler integration, script updates, and testing. The overall design aligns with project constraints (single-process, real-time scheduling, token passing). However, some areas require deeper consideration of error handling, edge cases, and integration complexity.

### Strengths

- **Clear decomposition**: Plans are well-structured into 7 sub-plans with explicit dependencies.
- **Real-time scheduling preserved**: Thread priorities (SCHED_FIFO 99, 90, 95, 0) correctly maintained.
- **Backward compatibility**: Legacy mode option in scripts eases migration.
- **Comprehensive testing**: Plan 05-06 includes unit and integration tests.
- **Dynamic watermark integration**: MCS-based rate mapping used for backpressure calculation.
- **Zero-polling唤醒**: Proper use of condition_variable (no usleep) in TX thread.

### Concerns

- **HIGH**: Error handling gaps — many functions lack error recovery (e.g., pcap_inject failure, TUN creation failure).
- **HIGH**: Memory safety — raw pointers and manual memory management in some C++/C interop (zfex) risk leaks.
- **MEDIUM**: Integration complexity — merging three separate codebases (rx.cpp, tx.cpp, wfb_tun.c) may introduce subtle bugs.
- **MEDIUM**: Real-time priority inversion risk — SCHED_FIFO threads could starve CFS threads if not carefully designed.
- **LOW**: Hardcoded constants (e.g., kFEC_N=12, kFEC_K=8) that should be configurable.
- **LOW**: Token expiration logic may have race conditions between RX and TX threads.

### Suggestions

- Add detailed error handling for all system calls (pcap_open, ioctl, socket, etc.) with retry logic.
- Use smart pointers (std::unique_ptr) for resources like pcap_t, zfex_code.
- Implement a configuration struct for FEC parameters (N, K) and make them command-line options.
- Add thread safety annotations (e.g., `[[guarded_by(mtx)]]`) to clarify locking discipline.
- Consider using a state machine for the TX thread to handle Token expiration more robustly.
- Include integration stress tests that simulate packet loss and variable latency.

### Risk Assessment

**MEDIUM** — The phase plans are technically sound but carry integration risk due to the complexity of merging three previously independent processes. Missing error handling could lead to silent failures in production. However, the phased approach (6 sub-plans with dependencies) mitigates risk by allowing incremental validation.

## 核心发现

### 技术栈

**主要编程语言**: C++ (C++11 及以上)
- **原因**: 性能要求高，需要精确的实时调度和零延迟内存管理
- **关键特性**: 无 GC，完全可控的内存管理，支持原子操作和条件变量

**系统级编程**:
- **Linux Kernel API**: TUN/TAP 设备、Raw Sockets、实时调度（SCHED_FIFO）、高精度定时（timerfd）
- **POSIX 线程库**: std::thread, std::mutex, std::condition_variable, std::atomic
- **网络库**: libpcap（Monitor 模式）、标准套接字 API

**关键数据结构**:
- TokenFrame（8 字节超轻量级控制帧）
- ThreadSafeQueue（10000 包静态分配环形队列）
- 全局原子变量（can_send, token_expire_time_ms, dynamic_queue_limit）

### 关键特性

**基础准入特性 (Table Stakes)**:
1. **MAC 控制帧**: Token Passing 核心信令，跳过 FEC，使用 Radiotap 最低速率模板，三连发击穿
2. **三态状态机**: 枯竭与睡眠、唤醒与需量拟合、高负载贪突发
3. **动态水位线计算**: 基于物理速率和授权时间动态计算，1.5 倍底仓容忍抖动
4. **进程合并**: wfb_tun 和 wfb_tx 合并为单进程，避免 UDP 通信环路丢包
5. **实时调度**: 控制面 SCHED_FIFO 99、发射端 SCHED_FIFO 90、读取线程普通优先级
6. **零轮询唤醒**: 使用 std::condition_variable 替代 usleep()，消除 1-5ms 毛刺
7. **快速丢包反压**: 极小 TUN 池（txqueuelen 100）+ 用户态浅水池
8. **下行呼吸周期**: 800ms 呼出 + 200ms 吸气，NACK 窗口 25ms/节点
9. **空闲巡逻模式**: 长间隔（100ms）单节点探询，消除探询风暴
10. **混合交织调度**: AQ 队列 N 次服务穿插 1 次 SQ 探询（N=3-5）

**差化特性 (Differentiators)**:
1. **动态水位线计算**: MCS 查找表映射，实时物理速率自适应
2. **三态状态机跃迁引擎**: Watchdog 动量拦截、加法扩窗、需量拟合
3. **消除探询风暴**: 长间隔空闲巡逻，频谱占用 < 15%
4. **混合交织调度保护**: 活跃节点带宽保障，探询开销 < 2%
5. **微超发浅缓冲 + 快速丢包**: 秒级排队延迟，消灭超时假死
6. **控制帧 Radiotap 模板分流**: 双模板设计，零延迟切换，最优穿透力

### 架构设计

**三层网络栈**:
1. **应用层**: UFTP 零感知，UDP/IP 通信，微超发限速（`-R` 参数）
2. **操作系统层**: Linux TUN 虚拟网卡，极小水池防抖动（txqueuelen 100）
3. **MAC/物理层**: 魔改 wfb-ng 单进程架构，无碰撞调度中心控制

**核心线程**:
- **TUN 读取线程（装弹手）**: 普通优先级，CPU 密集型 FEC 编码
- **控制面接收线程（哨兵）**: 最高实时优先级（SCHED_FIFO 99），poll 极速抽干队列
- **空口发射线程（射手）**: 次高实时优先级（SCHED_FIFO 90），condition_variable 零轮询唤醒

**数据流**:
```
UFTP → TUN 设备 → TUN 读取线程 → FEC 编码 → 物理内存池
  ↓                                            ↓
Token 响应 ← 控制面接收 ← 空口接收 ← WiFi Monitor 模式
  ↓
空口发射线程 → pcap_inject → 空口发射
```

### 关键参数

**MCS 速率映射表**（动态水位线计算核心）:
- MCS 0 (20MHz): 6Mbps
- MCS 4 (20MHz): 24Mbps
- MCS 8 (40MHz): 144Mbps
- **要求**: C++ 代码必须解析启动参数，通过查找表映射实时物理速率

**时间参数**（毫秒级容错）:
- MIN_WINDOW: 50ms（覆盖 OS 调度抖动 20ms + 侦听底线 30ms）
- MAX_WINDOW: 300ms (10 节点) / 600ms (5 节点)
- STEP_UP: 100ms（2-3 周期拉满占空比）
- ELASTIC_MARGIN: 50ms（包容系统抖动和投递毛刺）
- MICRO_PROBE_DURATION: 15ms（微探询）
- GUARD_INTERVAL: 5ms（节点切换保护）
- IDLE_PATROL_INTERVAL: 100ms（消除探询风暴）

### 主要陷阱

**P0 级别（致命）**:
1. **双进程通信环路数据静默丢失**: 必须合并 wfb_tun 和 wfb_tx 进程
2. **水位线硬编码导致 Bufferbloat**: 必须使用 MCS 动态映射表
3. **实时调度未提权导致毛刺失控**: 关键线程必须 SCHED_FIFO
4. **usleep() 导致灾难性轮询延迟**: 必须使用 condition_variable
5. **TUN 队列设置过大导致反压失效**: 必须设置 txqueuelen 100
9. **MAX_WINDOW 计算不当导致 UFTP 超时**: 必须满足 `(N-1) × MAX_WINDOW < GRTT`
10. **节点切换无保护间隔导致空中撞车**: 必须插入 5ms 保护间隔

**P1 级别（重要）**:
6. **控制帧 FEC 编码导致延迟积压**: 控制帧必须跳过 FEC
7. **固定周期探询风暴**: 必须长间隔空闲巡逻 + 混合交织调度
8. **Watchdog 未保护导致丢包误判**: 必须连续 3 次静默才判定断流

## 架构洞察

### 核心创新点

1. **中心强管控的 Token Passing** 彻底消灭半双工环境下的射频碰撞
2. **微超发浅缓冲 + 快速丢包** 将最坏排队延迟锁死在秒级，消灭 Bufferbloat
3. **三态状态机动态扩窗** 平衡 OS 调度抖动容忍与大模型突发吞吐需求
4. **毫秒级容错的保护间隔** 避免微秒级同步的严苛要求，普通 Linux 也能跑

### 与传统方案对比

| 方案 | 碰撞率 | 延迟控制 | 实现复杂度 | 实时性要求 |
|------|--------|----------|------------|------------|
| **Token Passing（本方案）** | 0% | 秒级可控 | 中 | 毫秒级 |
| TDMA（宏观同步） | < 1% | 微秒级 | 极高 | 微秒级 |
| CSMA/CA（分布式协商） | 10-30% | 秒级不可控 | 低 | 无 |

**结论**: Token Passing 在平衡复杂度、实时性要求和性能方面最优。

### 应用层零感知设计

**UFTP 无需修改**: 继续使用标准 UDP/IP，通过以下机制保持透明：
- 微超发限速（`-R` 参数）确保有时间片到来时有包可发
- 极小 TUN 池快速丢包反压，UFTP 收到 ENOBUFS 拥塞信号
- 适度调大超时阈值（GRTT），容忍秒级合理波动

**集成方式**: UFTP 发送流量到 TUN 设备，底层自动接管调度和传输。

## 结论

本架构通过 **中心强管控的 Token Passing** 机制，彻底消灭半双工无线环境下的射频碰撞，结合 **微超发浅缓冲 + 快速丢包反压**，将最坏排队延迟锁死在秒级，消除 Bufferbloat。**三态状态机动态扩窗** 平衡 OS 调度抖动容忍与大模型突发吞吐需求，**毫秒级容错的保护间隔** 避免微秒级同步的严苛要求。

系统严格分为三层（应用层、操作系统层、MAC/物理层），各司其职，严禁跨层耦合。合并 wfb_tun 和 wfb_tx 进程，使用动态水位线产生基于丢包的物理背压，对应用层 UFTP 保持绝对透明。

**预期成果**:
- 碰撞率: 0%
- 有效带宽利用率: > 90%
- 控制帧响应延迟: < 10ms
- 数据包排队延迟: < 2s
- 频谱占用（全员深睡）: < 15%

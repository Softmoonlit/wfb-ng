# Features: 联邦学习无线空口底层传输架构

**定义日期**: 2026-04-22

## 基础准入特性 (Table Stakes)

这些功能是联邦学习无线空口传输的最低要求，缺少任何一个都会导致系统无法正常工作。

### 控制平面
- **MAC 控制帧**: Token Passing 的核心信令机制
  - 包含魔数、目标节点、授权时间片、序列号
  - 跳过 FEC，使用 Radiotap 最低速率模板
  - 三连发击穿确保可靠性
- **序列号管理**: 全局单调递增序号，防止旧包诈尸
- **节点切换保护间隔**: 5ms 保护间隔，避免硬件队列未排空导致的空中撞车

### 调度器（服务端）
- **三态状态机**:
  - 状态 I: 枯竭与睡眠（Zero-Payload Drop）- Watchdog 防误判
  - 状态 II: 唤醒与需量拟合（Demand-Fitting）
  - 状态 III: 高负载贪突发（Additive Increase）
- **下行广播"呼吸"周期**:
  - 呼出 (800ms): 服务器全速下发模型
  - 吸气 (200ms): 服务器暂停，依次分配 NACK 回传窗口
- **空闲巡逻模式**: SQ 队列单节点探询，长间隔（100ms）消除探询风暴
- **混合交织调度**: AQ 队列 N 次服务穿插 1 次 SQ 探询（N=3-5）
- **自适应弹性扩窗 (AEW)**:
  - MIN_WINDOW = 50ms（底线）
  - MAX_WINDOW = 300ms（10 节点）/ 600ms（5 节点）
  - STEP_UP = 100ms（进攻扩窗步长）
  - ELASTIC_MARGIN = 50ms（弹性余量）

### 缓冲与流控（客户端+服务端）
- **物理内存池**: 静态分配环形队列（10000 包，约 15MB），杜绝内存溢出
- **动态逻辑水位线**: 基于物理速率和授权时间动态计算
  - 客户端: `limit = ceil((R_phy × duration_ms × 1.5) / (8 × MTU × 1000))`
  - 服务端: `limit = ceil((R_phy × Inhale_Time × 1.5) / (8 × MTU))`
- **快速丢包反压**: 水位线触发 TUN 读取停顿，导致 TUN 队列溢出丢包（ENOBUFS）
- **极小 TUN 池**: `txqueuelen 100`，确保快速背压传递

### 进程架构
- **wfb_tun 和 wfb_tx 合并**: 单进程内多线程，避免 UDP 通信环路丢包
- **控制/数据面分流**: 高速路由分发，独立线程处理
  - 控制面 (rx_demux_thread): 最高实时优先级（SCHED_FIFO 99）
  - 数据面读取 (tun_reader_thread): 普通优先级，CPU 密集型 FEC 编码
  - 发射端 (tx_main_loop): 次高实时优先级（SCHED_FIFO 90）
- **零轮询唤醒**: 使用 std::condition_variable 替代 usleep()，消除 1-5ms 毛刺

### 实时调度
- **SCHED_FIFO 优先级**:
  - 控制面线程: 优先级 99（最高）
  - 发射端线程: 优先级 90（次高）
  - 其他线程: 普通优先级（CFS）
- **高精度定时**: timerfd + 极短时间空转自旋，确保微秒级精度
- **保护间隔**: 5ms 强制间隔，覆盖硬件发射延迟

## 差异化特性 (Differentiators)

这些功能使系统在同类解决方案中脱颖而出，提供竞争优势。

### 动态水位线计算
- **基于实时物理速率**: MCS 查找表映射，而非硬编码
- **自适应背压**: 1.5 倍底仓容忍 OS 调度抖动和 UFTP 投递抖动
- **公式化设计**: 可预测、可验证、可调优

### 三态状态机跃迁引擎
- **Watchdog 动量拦截**: 防止单次丢包误判为断流
- **加法扩窗**: STEP_UP = 100ms，2-3 周期拉满占空比
- **需量拟合**: 紧贴实际消耗 + ELASTIC_MARGIN，保留爆发潜力

### 消除探询风暴
- **长间隔空闲巡逻**: IDLE_PATROL_INTERVAL = 100ms，将 50ms 风暴打散到 1150ms
- **低频谱占用**: 全员深睡时占用率 < 15%，极度省电
- **公平调度**: 即使睡眠节点也会被定期探询，避免饿死

### 混合交织调度保护
- **活跃节点带宽保障**: 每 N 次 AQ 服务穿插 1 次 SQ 探询
- **探询开销极低**: 15ms 探询占比 < 2%，对活跃节点无感
- **后台轮询完成**: 约 10.8 秒一轮，9 个睡眠节点悄无声息被探询

### 微超发浅缓冲 + 快速丢包
- **UFTP 微超发**: 限速设置为期望带宽 × 1.2，确保有时间片到来时有包可发
- **快速背压触发**: TUN 队列 100 个包溢出即丢，瞬间传递拥塞信号
- **零 Bufferbloat**: 最坏排队延迟锁死在秒级，消灭超时假死

### 控制帧 Radiotap 模板分流
- **双模板设计**: 一份高阶 MCS（业务数据），一份最低 MCS（控制帧）
- **零延迟切换**: 发射循环中判断帧类型直接拼接对应 Header
- **最优穿透力**: 最低 MCS 确保控制帧全网覆盖，无积压延迟

## 反模式 (Anti-Features)

这些功能被明确排除，不应在实现中出现。

### ❌ 分布式协商
- **排除原因**: 半双工无线环境，分布式协商会导致碰撞
- **替代方案**: 中心强管控的 Token Passing

### ❌ 无脑深水池缓冲
- **排除原因**: 无限积压导致 Bufferbloat，应用层超时假死
- **替代方案**: 微超发浅缓冲 + 快速丢包反压

### ❌ 固定周期集中探询
- **排除原因**: 50ms 风暴频谱占用高，浪费电池
- **替代方案**: 长间隔空闲巡逻 + 混合交织调度

### ❌ 控制帧 FEC 编码
- **排除原因**: FEC 组块积压导致延迟，实时性差
- **替代方案**: 三连发击穿 + 最低 MCS 速率

### ❌ 宏观同步 TDMA
- **排除原因**: 微秒级同步要求极高，Linux 非实时内核无法保障
- **替代方案**: 毫秒级容错的 Token Passing + 保护间隔

### ❌ Python/Go 等带 GC 的语言
- **排除原因**: GC 运行时抖动导致调度失准
- **替代方案**: 纯 C/C++ 零延迟

### ❌ 应用层感知底层调度
- **排除原因**: 跨层耦合导致灵活性差
- **替代方案**: UFTP 继续使用 UDP/IP，对底层零感知

## 场景覆盖

### 场景 1: 全员深睡
- **触发条件**: 所有客户端在训练，AQ 队列为空
- **调度行为**: 空闲巡逻模式，单节点 15ms 探询 + 100ms 间隔
- **预期结果**: 频谱占用 < 15%，极度省电，无探询风暴

### 场景 2: 单节点活跃
- **触发条件**: 1 个节点完成训练进入 AQ，9 个节点仍在 SQ
- **调度行为**: 混合交织 (N=4)，活跃节点连续 4 次大窗口 (累计 1200ms)
- **预期结果**: 活跃节点吞吐 > 98%，探询开销 < 2%

### 场景 3: 多节点并发
- **触发条件**: 3-5 个节点同时完成训练进入 AQ
- **调度行为**: AQ 队列轮询服务，SQ 探询穿插 (N=3-5)
- **预期结果**: 无碰撞，有效带宽利用率 > 90%，公平调度

### 场景 4: 下行大模型广播
- **触发条件**: 服务器向 10 个客户端下发 40MB 模型
- **调度行为**: 呼吸周期 (800ms 发 + 200ms 吸)，NACK 窗口 25ms/节点
- **预期结果**: 40MB 稳定传输，UFTP 无超时，NACK 可靠回传

### 场景 5: 动态信道质量变化
- **触发条件**: MCS 自动降级或升级，物理速率波动
- **调度行为**: 水位线公式动态调整，重新计算 limit
- **预期结果**: 水位线始终贴合物理速率，不超不满

- **下行广播效率**: 呼吸周期 overhead < 20%

- **碰撞率**: 0% (中心调度器彻底消除)

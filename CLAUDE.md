# Claude Code 项目指令

## 项目：联邦学习无线空口底层传输架构

**项目目标**: 基于 wfb-ng 改造为具有深缓冲区与动态时间配额轮询（Token Passing）的无碰撞传输架构，彻底解决联邦学习在半双工无线空口中的射频碰撞与拥塞问题。

**核心价值**: 基于深缓冲区与动态时间配额轮调度的无碰撞传输架构，彻底消灭射频碰撞并逼近物理极限带宽利用率，确保联邦学习 40MB 模型参数稳定传输。

**技术栈**: C++ (C++11 及以上), Linux (TUN 设备, 实时调度 SCHED_FIFO, poll/epoll), libpcap/Raw Sockets (WiFi Monitor Mode)。

**当前阶段**: Phase 1 - 基础框架与数据结构

## 项目架构要求

### 三层网络栈架构

系统严格分为三层，各司其职，严禁跨层耦合：

**1. 应用层：UFTP 零感知**
- 职责：按序、可靠地组播下发/收集 40MB 模型文件
- 要求：无需修改 UFTP 源码，使用标准 UDP/IP 协议
- 配置：微超发限速 `-R` 参数，适度调大 GRTT 超时阈值

**2. 操作系统层：Linux TUN 极小水池**
- 职责：进程内核交互的微减震器，确保存取平滑，传递满载反压
- 要求：TUN 队列设置 txqueuelen 100
- 原理：快速丢包反压，向 UFTP 返回 ENOBUFS 拥塞信号

**3. MAC/物理层：魔改 wfb-ng**
- 职责：彻底接管物理发射权，无 Token 不发射
- 要求：合并 wfb_tun 和 wfb_tx 进程，单进程内多线程
- 调度：中心强管控的 Token Passing，三态状态机动态扩窗

### 核心设计原则

**必须遵守的关键原则**：

1. **合并进程架构**: 严禁使用独立进程通过 UDP 通信，必须在单进程内实现 TUN 读取和空口发射
2. **动态水位线计算**: 严禁硬编码水位线，必须基于 MCS 速率映射表动态计算
3. **实时调度提权**: 关键线程必须提权至 SCHED_FIFO（控制面 99、发射端 90）
4. **零轮询唤醒机制**: 严禁使用 usleep()，必须使用 std::condition_variable
5. **极小 TUN 池**: 必须设置 txqueuelen 100，触发快速丢包反压
6. **控制帧跳过 FEC**: 严禁对控制帧进行 FEC 编码，避免延迟积压
7. **Watchdog 保护**: 状态 I 必须连续 3 次静默才判定断流，单次丢包保持窗口
8. **节点切换保护**: 必须插入 5ms 保护间隔，避免空中撞车
9. **应用层零感知**: UFTP 继续使用 UDP/IP，对底层调度完全透明

## 核心陷阱防范

在编写代码时，必须时刻注意以下 10 个致命陷阱：

### P0 级别（致命）

1. **双进程通信环路数据静默丢失**: 必须合并 wfb_tun 和 wfb_tx 进程
2. **水位线硬编码导致 Bufferbloat**: 必须使用 MCS 动态映射表
3. **实时调度未提权导致毛刺失控**: 关键线程必须 SCHED_FIFO
4. **usleep() 导致灾难性轮询延迟**: 必须使用 condition_variable
5. **TUN 队列设置过大导致反压失效**: 必须设置 txqueuelen 100
9. **MAX_WINDOW 计算不当导致 UFTP 超时**: 必须满足 `(N-1) × MAX_WINDOW < GRTT`
10. **节点切换无保护间隔导致空中撞车**: 必须插入 5ms 保护间隔

### P1 级别（重要）

6. **控制帧 FEC 编码导致延迟积压**: 控制帧必须跳过 FEC
7. **固定周期探询风暴**: 必须长间隔空闲巡逻 + 混合交织调度
8. **Watchdog 未保护导致丢包误判**: 必须连续 3 次静默才判定断流

## 关键参数配置

### 时间参数（毫秒级容错）

| 参数 | 值 | 用途 |
|------|-----|------|
| MIN_WINDOW | 50ms | 覆盖 OS 调度抖动（20ms）+ 侦听底线（30ms） |
| MAX_WINDOW | 300ms (10 节点) / 600ms (5 节点) | 确保 (N-1)×MAX_WINDOW < UFTP 断连超时 |
| STEP_UP | 100ms | 2-3 周期拉满占空比 |
| ELASTIC_MARGIN | 50ms | 包容系统抖动和投递毛刺 |
| MICRO_PROBE_DURATION | 15ms | 覆盖调度抖动 + 网卡延迟 |
| GUARD_INTERVAL | 5ms | 节点切换保护，避免空中撞车 |
| IDLE_PATROL_INTERVAL | 100ms | 消除探询风暴，低频谱占用 |

## 开发流程

### TDD 驱动开发

1. **编写失败的测试**: 针对每个核心功能
2. **运行测试验证失败**: 确保测试有效
3. **编写最小实现**: 通过测试的最简单代码
4. **验证测试通过**: 确保实现正确
5. **提交代码**: 原子提交，粒度细

### 代码质量标准

- **编码规范**: 遵循 Google C++ Style Guide
- **代码注释**: 所有函数和关键逻辑添加中文注释
- **静态分析**: 通过 Cppcheck, clang-tidy 检查
- **内存安全**: 使用 RAII、智能指针（现代 C++）
- **线程安全**: 多线程访问必须加锁或使用原子操作

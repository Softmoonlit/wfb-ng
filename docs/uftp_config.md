# UFTP 配置指南

本文档提供 UFTP 应用层的限速配置和超时调整指南，确保应用层与底层 Token Passing 调度正确集成，实现零感知适配。

---

## 1. 限速公式推导

### 1.1 核心公式

UFTP 微超发限速的核心公式：

```
rate_limit = (R_phy / N) × 1.2
```

其中：
- `R_phy`：物理层速率（Mbps），从 MCS 映射表获取
- `N`：节点总数（客户端数量）
- `1.2`：微超发系数，允许适度竞争以提高频谱利用率

### 1.2 UFTP `-R` 参数计算

UFTP 的 `-R` 参数单位为 KB/s，计算公式：

```
-R 值 = rate_limit × 1000 / 8 = (R_phy / N) × 1.2 × 1000 / 8
```

### 1.3 设计原理

**为什么需要微超发？**

1. **避免过度保守**：精确限速可能导致频谱空闲，1.2 倍系数允许适度竞争
2. **容忍调度抖动**：Token Passing 引入秒级轮询延迟，微超发确保队列不空
3. **快速反压机制**：TUN 设备 txqueuelen=100 触发快速丢包，向 UFTP 返回 ENOBUFS 拥塞信号

**为什么不超发更多？**

1. **反压失效风险**：过度超发导致 TUN 队列积压，延迟失控
2. **缓冲膨胀**：用户态队列超出水位线，内存压力增大
3. **应用层超时**：排队延迟超过 GRTT 触发 UFTP 重传风暴

## 4. UFTP 命令行参数说明

### 4.1 限速参数 `-R`

```
-R rate_limit
```

设置发送速率限制，单位为 KB/s。这是最关键的参数，用于控制 UFTP 发送速率，避免超过底层 Token Passing 调度能力。

**计算方法：**
```bash
# 1. 确定 MCS 和频宽
# 2. 从 MCS 映射表获取物理速率 R_phy
# 3. 确定节点数 N
# 4. 计算 -R 值
-R 值 = (R_phy / N) × 1.2 × 1000 / 8
```

### 4.2 其他关键参数

| 参数 | 说明 | 建议值 |
|------|------|--------|
| `-g group` | 组播组地址 | 如 224.1.1.1 |
| `-p port` | UDP 端口 | 如 1042 |
| `-T ttl` | 组播 TTL | 5-10 |
| `-r retries` | 重传次数 | 5-10（建议调大） |
| `-t timeout` | 超时时间 | 建议适度调大 |

## 5. 超时调整建议

### 5.1 GRTT（Group Round-Trip Time）

**建议值：5-10 秒**

**原因：**
1. Token Passing 调度引入秒级轮询延迟
2. 每个节点需等待 Token 授权才能发送
3. 最大等待时间 = (N-1) × MAX_WINDOW
4. 需要容忍合理的调度波动

**计算依据：**
```
单轮最大延迟 = (N - 1) × MAX_WINDOW

例如：10 节点，MAX_WINDOW = 300ms
单轮最大延迟 = 9 × 300ms = 2.7 秒

建议 GRTT > 单轮最大延迟 × 2 = 5.4 秒
```

### 5.2 重传次数

**建议值：5-10 次**

**原因：**
1. 无线环境存在随机丢包
2. Token 丢失需要等待下一轮授权
3. 控制帧三连发击穿机制提高可靠性
4. 调高重传次数增加容错能力

### 5.3 UFTP 断连超时

**关键公式：**
```
(N - 1) × MAX_WINDOW < UFTP 断连超时
```

**配置建议：**
- 10 节点：MAX_WINDOW = 300ms，断连超时 > 3 秒
- 5 节点：MAX_WINDOW = 600ms，断连超时 > 2.5 秒

## 6. 完整配置示例

### 场景 1：10 节点，MCS 6 (54 Mbps)，20MHz 频宽

```bash
# 限速计算：(54 / 10) × 1.2 = 6.48 Mbps = 810 KB/s
# GRTT 建议：5-10 秒
# 重传次数：5-8 次

uftp -R 810 -g 224.1.1.1 -p 1042 -T 10 -r 5 model.bin
```

### 场景 2：5 节点，MCS 8 (144 Mbps)，40MHz 频宽

```bash
# 限速计算：(144 / 5) × 1.2 = 34.56 Mbps = 4320 KB/s
# GRTT 建议：5-8 秒
# 重传次数：5-8 次

uftp -R 4320 -g 224.1.1.1 -p 1042 -T 5 -r 8 model.bin
```

### 场景 3：10 节点，MCS 0 (6 Mbps)，20MHz 频宽（最差情况）

```bash
# 限速计算：(6 / 10) × 1.2 = 0.72 Mbps = 90 KB/s
# GRTT 建议：8-10 秒（低速链路需要更长超时）
# 重传次数：8-10 次

uftp -R 90 -g 224.1.1.1 -p 1042 -T 10 -r 10 model.bin
```

### 场景 4：5 节点，MCS 4 (24 Mbps)，20MHz 频宽

```bash

# GRTT 建议：5-8 秒
# 重传次数：5-8 次

uftp -R 720 -g 224.1.1.1 -p 1042 -T 5 -r 6 model.bin
```

## 7. 底层原理阐述

### 7.1 三层架构

系统严格分为三层：

1. **应用层（UFTP）**
   - 使用标准 UDP/IP 协议
   - 对底层调度零感知
   - 微超发限速控制发送速率

2. **操作系统层（Linux TUN）**
   - 极小水池（txqueuelen=100）
   - 快速丢包反压
   - 向应用层返回 ENOBUFS 拥塞信号

3. **MAC/物理层（魔改 wfb-ng）**
   - Token Passing 中心调度
   - 无碰撞传输
   - 动态水位线控制队列深度

### 7.2 反压机制

```
UFTP 发送过快
    ↓
TUN 队列积压（txqueuelen=100）
    ↓
内核队列溢出，返回 ENOBUFS
    ↓
UFTP 收到拥塞信号，退避降速
    ↓
用户态队列低于动态水位线
    ↓
TUN 读取线程恢复读取
```

### 7.3 动态水位线

水位线公式：
```
limit = ceil((R_phy × duration_ms × 1.5) / (8 × MTU × 1000))
```

其中：
- `R_phy`：物理速率（bps）
- `duration_ms`：Token 授权时间（ms）
- `MTU`：最大传输单元（通常 1500 字节）
- `1.5`：底仓系数，容忍调度抖动

### 7.4 Token Passing 调度

**核心原则：无 Token 不发射**

1. 服务器作为中心调度器，分配发射时间片
2. 客户端收到 Token 后获得授权时间窗口
3. 窗口内独占空口，无碰撞传输
4. 窗口结束，Token 失效，等待下一轮

**保护机制：**
- 5ms 保护间隔，避免节点切换时空中撞车
- Watchdog 保护，连续 3 次静默才判定断流
- 控制帧三连发击穿，提高可靠性

### 7.5 呼吸周期

服务器下行广播采用"呼吸"模式：

```
呼出 (800ms): 服务器全速下发 40MB 模型
    ↓
吸气 (200ms): 服务器暂停，分配 NACK 回传窗口
    ↓
循环执行直至 UFTP 确认 DONE
```

## 8. 常见问题

### Q1: UFTP 超时怎么办？

**检查项：**
1. 确认 `-R` 参数配置正确，不超过建议值
2. 确认 GRTT 设置足够大（5-10 秒）
3. 确认重传次数设置足够（5-10 次）
4. 检查底层 Token Passing 调度是否正常
5. 检查无线链路质量

### Q2: 吞吐量低于预期？

**检查项：**
1. 确认 MCS 配置与实际链路匹配
2. 检查信道是否有干扰
3. 确认 TUN 设备 txqueuelen=100
4. 检查动态水位线是否过高限制队列深度

### Q3: 队列积压过高？

**检查项：**
1. 确认 `-R` 参数不超过建议值
2. 检查 Token 授权时间是否足够
3. 确认 MCS 映射正确，物理速率估算准确


```bash

```

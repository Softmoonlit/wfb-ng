## 开场白（30 秒）

**讲解词**：
各位专家好，今天我汇报的题目是"联邦学习无线空口底层传输架构设计"。这个方案的核心是基于 Token Passing 机制，彻底解决联邦学习在半双工无线空口中的射频碰撞问题。

我们的目标是实现**零碰撞、高带宽利用率、对应用层零感知**的传输架构。

## 第一部分：问题背景与挑战（6 分钟）

### Slide 2: 联邦学习场景与挑战（1 分钟）

**讲解词**：
首先，让我们了解一下应用场景。联邦学习是一种分布式机器学习范式，我们面对的是这样一个场景：

**一个服务器，十个客户端**，每个客户端在本地训练模型，然后周期性地把 40MB 的模型参数上传到服务器进行聚合，服务器再把更新后的模型下发给客户端。

这里有一个关键问题：联邦学习有**三大特征**。

**第一，非对称性**。这是一个典型的星型拓扑，一个服务器对多个客户端，数据流向是不对称的。

**第二，突发性**。40MB 的模型文件不是连续小包传输，而是突发式的大文件传输。这对我们的网络协议设计提出了很高的要求。

**第三，异步性**。每个客户端本地训练的时间是随机的，有的可能训练 5 分钟，有的可能训练 10 分钟。也就是说，我们无法预测哪个客户端会在什么时候完成训练，准备好发送数据。

这三大特征，直接导致了我们接下来要讲的核心问题。

---

### Slide 3: 无线空口的特殊性（1 分钟）

**讲解词**：
要理解这个问题，我们需要先了解无线空口的特殊性。

我们的系统运行在 **WiFi Monitor 模式**下。这是一种特殊的模式，它绕过了内核网络栈，直接访问物理层，可以自定义数据帧格式，这对于实现我们的 Token Passing 机制是必要的。

但是，Monitor 模式有一个致命的缺陷：**它没有 CSMA/CA 冲突避免机制**。

CSMA/CA 是什么？简单来说，它是 WiFi 的"礼貌协议"。在正常模式下，每个设备发送前都会先"听一听"信道是否空闲，如果空闲才发送。但 Monitor 模式把这个机制关掉了，变成了"想发就发"。

这就像一个没有交通信号灯的十字路口：每辆车都不看路况，直接冲过去，结果就是——撞车。

这就引出了我们的核心问题：**射频碰撞**。

---

### Slide 4: 射频碰撞问题演示（1.5 分钟）

**讲解词**：
什么是射频碰撞？

简单来说，当多个节点同时发送数据时，它们的无线电波会在空中叠加，导致信号失真，接收端无法正确解码。这就是物理层的碰撞。

我们在实际测试中发现，在没有任何调度机制的情况下，当 10 个节点同时传输时，**碰撞率高达 30-50%**。这意味着什么？接近一半的数据包都丢失了。

更糟糕的是，这会引发连锁反应：

应用层检测到数据丢失，会触发超时重传。但重传的数据包又会和新的数据包碰撞，导致更严重的拥塞。最终，**有效带宽利用率降到 20% 以下**。

对于联邦学习来说，这意味着什么？训练中断，模型无法聚合，整个系统陷入瘫痪。

这里我用一个类比：就像一个没有红绿灯的十字路口，车流高峰期时，所有车都想过路口，结果全都堵在中间，动弹不得。

---

### Slide 5: 现有方案的局限性（1.5 分钟）

**讲解词**：
那么，现有的解决方案是什么呢？为什么它们不能解决问题？

**方案一：分布式协商（RTS/CTS）**。

这是 WiFi 标准中的机制：发送方先发一个 RTS 帧，接收方回复 CTS，确认信道空闲后再发送。但这个方案的问题是：**每次发送前都要握手，开销太大**。在我们的测试中，有效吞吐量不到 50%。对于 40MB 的模型传输，这个开销是无法接受的。

**方案二：静态 TDMA 时隙分配**。

这是最直观的方案：给每个节点分配固定的时隙，比如每个节点 30ms，轮流发送。这个方案的优点是简单，无碰撞。

但问题在于：**无法适应突发流量**。联邦学习的异步性意味着大部分时候大部分节点都没有数据要发，这些时隙就被浪费了。而真正有数据要发的节点，又受限于固定时隙，无法充分利用带宽。

**方案三：应用层限速**。

这个方案是在应用层（比如 UFTP）限制发送速率，避免网络拥塞。但这个方案**治标不治本**，无法从根本上解决物理层碰撞的问题。

所以，我们需要一个新的机制。

---

### Slide 6: 核心挑战总结（1 分钟）

**讲解词**：
总结一下，我们面临三大核心挑战：

**挑战一：如何在物理层彻底消灭碰撞？**

分布式协商开销大，静态时隙浪费严重。我们需要一个机制，能够集中控制发射权，从根本上避免碰撞。

**挑战二：如何对应用层（UFTP）保持透明？**

我们不能修改 UFTP 的源码，它必须继续使用标准的 UDP/IP 协议。这是工程约束，也是设计原则。

**挑战三：如何适应联邦学习的突发流量特性？**

既要避免碰撞，又要充分利用带宽；既要公平，又要高效。这需要一个动态的、智能的调度机制。

答案是：**Token Passing 中心调度**。

---

## 第二部分：解决方案核心思想（3 分钟）

### Slide 7: Token Passing 核心思想（1.5 分钟）

**讲解词**：
Token Passing 的核心思想是：**服务器作为中心调度器，统一管理所有节点的发射权，实现无碰撞传输**。

具体来说，服务器维护一个节点列表，依次给每个节点授权发射：

服务器给某个节点发送一个 Token（令牌），这个 Token 包含目标节点 ID 和时间窗口大小。目标节点收到 Token 后，确认是自己，就在窗口内发送数据。

时间窗口到期后，服务器再给下一个节点发送 Token。任何时候，空中只有一个节点在发射，**彻底消灭了物理层碰撞**。

而且，时间窗口不是固定的。服务器会根据节点的实际数据量动态调整：如果节点有大量数据积压，就分配更大的窗口；如果节点队列空了，就分配小窗口甚至跳过。

这就是 Token Passing 的核心思想：**中心化的发射权控制 + 动态时间配额分配**。具体的技术细节，我会在第三部分沿数据流详细讲解。

---

### Slide 8: 三层架构设计（全局视图）（1.5 分钟）

**讲解词**：
为了实现 Token Passing，我们设计了一个三层架构。

**最上层是应用层，也就是 UFTP**。

UFTP 的职责是可靠地传输模型文件，它提供前向纠错（FEC）、包分片、乱序重排等功能。但关键是，**我们对 UFTP 是零感知的**。它不需要知道底层有 Token Passing，继续使用标准的 UDP/IP 协议。

**中间层是 OS 层，主要包括 TUN 设备和环形队列**。

TUN 设备是一个虚拟网络接口，它拦截 UFTP 发送的数据包，放入一个环形队列中。这个队列是一个"深缓冲区"，可以容纳 10000 个数据包。

**注意**：调度器不在这一层，而是在物理层。OS 层只负责缓冲和背压，不负责调度决策。

**最底层是物理层，也就是魔改后的 wfb-ng**。

这里实现了 Token Passing 的核心逻辑：
- **服务器端**：Token Passing 调度器，负责中心化发射权控制，决定哪个节点何时发送
- **客户端**：Token 解析与发射控制，收到 Token 后在授权窗口内发送数据
- **通用功能**：Monitor 模式帧注入，Radiotap 模板分流

三层各司其职，**应用层只管可靠传输，OS 层只管缓冲背压，物理层只管调度发射**。这种分离设计，使得我们可以独立优化每一层，而不会产生跨层耦合。

接下来，让我们沿着数据流的路径，详细讲解每个环节的技术实现。

## 第三部分：Token Passing 流程与关键技术（12 分钟）

### Slide 9: Token Passing 流程全景图（1 分钟）

**讲解词**：
第三部分我们聚焦于**客户端上传模型的 Token Passing 流程**。

服务器下发模型采用的是独立的"呼吸周期"策略，这里暂不展开。

**Token Passing 完整流程**：
```
┌─────────────────────────────────────────────────────────────────┐
│ 客户端：UFTP → TUN → 环形队列 → 等待 Token → 在窗口内发送 → 空口 │
│                              ↑                                  │
│ 服务器：调度决策 → 生成 Token ──────┘                            │
│         ↑                                                       │
│ 客户端：发送数据 → 服务器接收 → 确认/更新状态 → 发送下一个 Token │
└─────────────────────────────────────────────────────────────────┘
```

这个流程涉及**服务器和客户端双方**，我们实现了 **14 个关键技术**。

让我们沿着数据流，从客户端的数据注入开始，逐一讲解。

### Slide 10: 应用层数据注入 → TUN 极小水池【客户端】（1 分钟）

**讲解词**：
首先是**应用层数据注入**。

UFTP 通过标准的 UDP socket 发送数据包，这些数据包会进入 Linux 内核的网络栈，然后被路由到 TUN 设备。

这里我们有一个关键设计：**关键技术 1：TUN 极小水池**。

正常情况下，TUN 设备的队列长度是 1000 个包。但我们把它设置为 **txqueuelen = 100**，只有默认值的十分之一。为什么？

为了触发**快速丢包反压**。

当调度器来不及处理数据包时，队列会快速填满（因为容量很小）。此时，新到达的数据包会被丢弃，内核会返回 ENOBUFS 错误给 UFTP。

UFTP 收到这个错误后，会暂停发送，等待网络恢复。这就是"背压"：下游（调度器）处理不过来，就让上游（UFTP）慢一点。

如果队列太大（1000），数据包会在队列里排队很久，UFTP 以为发送成功了，但实际上数据包还在队列里等待。这会导致应用层超时假死。

所以，**极小水池 = 快速反馈**。

---

### Slide 12: TUN 读取 → 环形队列深缓冲【客户端】（1.5 分钟）

**讲解词**：
接下来是**环形队列深缓冲**。

我们有一个 TUN 读取线程，负责从 TUN 设备读取数据包，放入环形队列。

这里有两个关键技术：

**关键技术 2：线程安全环形队列**。

这个队列是一个环形缓冲区，容量是 10000 个数据包。我们采用**静态分配**，在初始化时就分配好内存，运行时不再动态分配。这样可以避免内存碎片，提高性能。

队列是线程安全的，使用互斥锁保护。关键是：**当队列达到水位线时，停止从 TUN 读取**，而不是丢弃数据包。这样才能产生反压。

**关键技术 3：动态水位线机制**。

现在问题来了：队列容量是 10000，但不是等队列真的满了（10000）才停止读取。我们应该提前预警，比如队列到了 8000 时就开始放慢速度。

这个"预警线"就是**水位线**。但水位线应该是多少？不能是固定值，因为不同的物理速率，发送 8000 个包所需的时间是不同的。

所以，我们动态计算水位线：
```
limit = (PHY_Rate × duration / MTU) × 1.5
```
其中 PHY_Rate 是物理层速率（比如 54Mbps），duration 是时间窗口（比如 200ms），MTU 是最大传输单元（1500 字节），1.5 是安全系数。

水位线是动态的，因为物理速率会变化（MCS 0 是 6Mbps，MCS 8 是 144Mbps）。如果硬编码水位线，要么太高（浪费内存），要么太低（频繁触发反压）。

---

### Slide 13: 调度器决策 → 三态状态机调度【服务器】（2 分钟）

**讲解词**：
现在数据包在队列里了，接下来是**调度器决策**。

在讲调度逻辑之前，先说一个关键的系统设计：**实时调度提权**。

**关键技术 4：实时调度提权**。

Linux 默认的 CFS 调度器有 20ms 级别的调度抖动，对于毫秒级精度的 Token Passing 是不可接受的。

我们把关键线程提权到 **SCHED_FIFO 实时优先级**：
- **控制面线程**（收包）：优先级 **99**（最高），确保 Token 不丢失
- **发射端线程**：优先级 **90**，确保空口发射及时
- **TUN 读取线程**：**普通优先级**，不做提权（因为 FEC 编码是 CPU 密集型，设为实时会死锁）

这样，关键路径上的延迟可控在微秒级，而 FEC 编码的抖动被深缓冲吸收。

---

现在看**关键技术 5：三态状态机调度**。

调度器是一个独立的线程，负责决定：**现在该让哪个节点发？发多久？**

每个节点都有一个状态，可以是：

**状态 I：睡眠态**。

如果节点连续 3 次被分配了时间窗口，但都没有发送数据（队列为空），我们就认为这个节点进入了"睡眠"状态，被迁移到 SQ（睡眠队列）。

当调度器从 SQ 探询睡眠节点时，使用**微探询时长 15ms**（MICRO_PROBE_DURATION），而不是正常窗口。这样既给节点发送机会，又避免浪费太多空口资源。

**Watchdog 保护**：如果节点上一轮处于高负载状态，这次突然静默，大概率是 Token 在空中丢失。此时**保持原窗口，不自增沉默计数器，下一轮优先重发 Token**。只有**连续 3 次**静默才真正判定断流。

为什么要连续 3 次？因为单次静默就降级，可能会误判。这是"去抖动"机制。

**状态 II：需量拟合态**。

这是正常状态。节点有数据要发，窗口计算公式：
```
window = max(MIN, actual + MARGIN)
```
其中 MIN = 50ms，actual 是上轮实际发送时长，MARGIN = 50ms。

这确保了节点有基本的发送机会，同时适应实际需求。

**状态 III：激进扩窗态**。

如果节点有大量数据（队列积压），我们会激进扩窗：
```
window = min(MAX, allocated + STEP)
```
其中 MAX = 300ms，allocated 是上轮分配的窗口，STEP = 100ms。

如果节点持续有大量数据，窗口会快速从 50ms → 150ms → 250ms → 300ms，只用 3 轮就拉满。

这样设计的好处：**适应突发流量**。联邦学习传输 40MB 模型时，节点需要大窗口；传输完成后，窗口自动收缩，避免浪费。

除了三态状态机，我们还设计了**关键技术 6：AQ/SQ 双队列管理**。

**AQ（Active Queue，活跃队列）**：当前有数据要发的节点。
**SQ（Sleep Queue，睡眠队列）**：长时间静默的节点。

调度器优先服务 AQ 中的节点，但不会饿死 SQ。我们采用**混合交织调度**：

**正常模式（AQ 和 SQ 都有节点）**：
```
调度顺序：AQ → AQ → AQ → AQ → SQ → AQ → AQ → AQ → AQ → SQ → ...
          └─── 4 个活跃节点 ───┘ └─ 1 个睡眠节点 ─┘
```
每服务 4 次 AQ，穿插 1 次对 SQ 的微探询（15ms）。这样既保证活跃节点的吞吐量，又能及时发现睡眠节点的新数据。

**空闲巡逻模式（AQ 为空，只有 SQ）**：
```
调度顺序：SQ[0] → SQ[1] → SQ[2] → ... → SQ[N-1] → SQ[0] → ...
```
此时所有节点都在睡眠，调度器轮询 SQ 中的每个节点，用 15ms 微窗口探询。一旦某个节点有数据，立即迁移到 AQ，恢复正常窗口。

**举个例子**：假设有 10 个节点，其中节点 1、2、3 在 AQ（活跃），节点 4-10 在 SQ（睡眠）。

```
轮次 1: 节点 1 (AQ, 正常窗口)
轮次 2: 节点 2 (AQ, 正常窗口)
轮次 3: 节点 3 (AQ, 正常窗口)
轮次 4: 节点 1 (AQ, 正常窗口)  ← 达到 4 次
轮次 5: 节点 4 (SQ, 微探询 15ms) ← 穿插探询
轮次 6: 节点 2 (AQ, 正常窗口)  ← 重新计数
...
```

这样设计的好处：**活跃节点获得大部分带宽，睡眠节点也不会被饿死**。

### Slide 14: Token 生成 → 授权控制帧【服务器】（1.5 分钟）

**讲解词**：
调度器决定好下一个发射节点后，就要生成 **Token 控制帧**。

这就是**关键技术 7：MAC 控制帧设计**。

TokenFrame 是一个紧凑的结构体，只有 8 字节：
```c
struct TokenFrame {
  uint8_t magic;         // 魔数，标识帧类型（1 字节）
  uint8_t target_node;   // 目标节点 ID（1 字节）
  uint16_t window_size;  // 时间窗口大小，单位 ms（2 字节）
  uint32_t seq_num;      // 序列号，防重放（4 字节）
};
```

**紧凑对齐**：使用 `#pragma pack(push, 1)` 确保无填充字节，总共 8 字节。序列化时直接内存拷贝，零拷贝，高效。

**序列号**：这是关键。每个 Token 都有一个全球单调递增的序列号，用于：
1. 防重放攻击：恶意节点不能重复发送旧 Token。
2. 去重：配合三连发机制，接收端可以识别重复的 Token。

生成 Token 后，就要序列化并发送。但这里有一个问题：**要不要对 Token 进行 FEC 编码？**

答案是：**跳过 FEC**。这就是**关键技术 8**。

为什么？FEC（前向纠错）是把多个数据包编码成更多包，容忍丢包。但代价是：引入延迟。FEC 编码需要等待一组数据包到齐，然后计算，这会引入 3-5ms 的延迟。

对于控制帧来说，延迟是不可接受的。Token 是授权信号，节点收到 Token 才能发送，延迟 5ms 意味着浪费 5ms 的空口时间。

而且，Token 很小（8 字节），不需要分片，也不容易丢包（我们有三连发机制）。

所以，**控制帧跳过 FEC，降低延迟**。

### Slide 15: 空口发射 → Radiotap 模板与三连发【服务器】（1.5 分钟）

**讲解词**：
Token 生成后，服务器要将其发送到空口。这里涉及三个关键技术：

**关键技术 9：Radiotap 模板分流**。

服务器发送帧时，根据帧类型选择不同的 Radiotap 模板：

**控制帧模板：MCS 0，速率 6Mbps**。

为什么用最低速率？因为要确保**全网覆盖**。距离服务器远的节点，信号可能很弱，低速率调制（BPSK）抗干扰能力强，即使信号弱也能解码。

**数据帧模板：MCS 8，速率 144Mbps**。

为什么用最高速率？因为要**最大化吞吐量**。数据帧是实际承载数据的，用高速率可以发送更多数据。

这里的关键是：**发射时零延迟切换模板**。两套模板在初始化时就准备好了，发射时只需要选择指针，不需要重新构建。

**关键技术 10：三连发击穿机制**。

无线环境是不可靠的，丢包是常态。对于控制帧（Token），它丢失了怎么办？

如果节点没收到 Token，它就无法发送，服务器会等待超时，浪费一个时间窗口。

所以，我们采用**三连发**：每个 Token 连续发送 3 次。接收端通过序列号去重，只处理一次。

**关键技术 11：节点切换保护间隔**。

服务器在给节点 A 发送完 Token（三连发）后，要给节点 B 发送 Token。此时需要插入保护间隔。

为什么？因为射频芯片的硬件发射队列可能还有残留的数据包正在发送。如果立即发送下一个 Token，可能会在空中撞车。

所以，服务器在发送下一个 Token 前，会**插入 5ms 的保护间隔**，确保硬件队列完全排空。

这样，**Radiotap 分流 + 三连发 + 保护间隔**，确保了控制帧的可靠性、覆盖范围和无碰撞。

---

### Slide 16: 客户端接收 Token → 发射控制【客户端】（1 分钟）

**讲解词**：
Token 发送到空口后，客户端接收并处理。

**关键技术 12：零轮询唤醒机制**。

客户端的发射线程使用 `std::condition_variable` 等待 Token。收到 Token 且目标是自己时，立即唤醒发射线程；无 Token 时持续休眠，不占用 CPU。

这样实现了**微秒级响应**，同时避免了 CPU 空转。

客户端收到 Token 后，解析 target_node 字段：
- 如果是自己 → 在窗口内发送数据
- 如果不是自己 → 保持静默

---

### Slide 17: 接收端处理 → 状态反馈与窗口调整【客户端】（1 分钟）

**讲解词**：
发送完毕后，客户端会在下一个周期收到新的 Token。服务器会根据客户端实际发送的数据量，调整窗口大小（三态状态机）。

这里有一个特殊情况：**所有节点都静默**怎么办？

如果所有节点的队列都空了，没有活跃节点，服务器该怎么办？

答案是：**关键技术 13：空闲巡逻模式**。

服务器进入"空闲巡逻模式"，从 SQ 队列轮询睡眠节点。**注意**：不是连续探询，而是**探询一个节点后，静默 100ms，再探询下一个**。

```
探询节点 1 (15ms) → 静默 100ms → 探询节点 2 (15ms) → 静默 100ms → ...
```

10 个节点轮询一遍需要：`10 × (15ms + 100ms) = 1150ms`。

**效果**：频谱占用 < 15%。对于动辄几分钟的联邦学习训练，晚唤醒 1-2 秒毫无影响，但频谱底噪极大降低。

---

### Slide 18: Token Passing 流程回顾（1 分钟）

**讲解词**：
现在，让我们回顾一下 Token Passing 的完整流程：

```
客户端数据流：UFTP → TUN（极小水池）→ 环形队列 → 水位线 → 等待 Token
                              ↓
服务器 Token 流：三态调度 → AQ/SQ 管理 → Token 生成 → 跳过 FEC →
                 Radiotap 分流 → 三连发 → 保护间隔 → 空口
                              ↓
客户端发射：收到 Token → 零轮询唤醒 → 在窗口内发送数据 → 空口
                              ↓
服务器确认：接收数据 → 更新状态 → 发送下一个 Token
```

这里有 **13 个关键技术**，每个技术都是解决一个具体问题：

1. TUN 极小水池：快速丢包反压
2. 环形队列：深缓冲，无动态内存分配
3. 动态水位线：适应不同 MCS 速率
4. **实时调度提权**：控制面 99，发射端 90，TUN 读取普通
5. 三态状态机：动态时间配额分配
6. AQ/SQ 管理：公平调度，不饿死睡眠节点
7. Token 生成：紧凑控制帧，零拷贝
8. 跳过 FEC：降低控制帧延迟
9. Radiotap 分流：控制帧全覆盖，数据帧高吞吐
10. 三连发：提高控制帧可靠性
11. 保护间隔：避免节点切换碰撞
12. 零轮询唤醒：微秒级响应
13. **空闲巡逻**：探询间隔 100ms，频谱占用 < 15%

**核心观点**：这不是 14 个孤立的技术，而是一个系统性的解决方案，每个技术都是为了解决一个具体问题，它们相互协作，共同实现了**零碰撞、高带宽利用率、零感知**的目标。

## 第四部分：总结与展望（3 分钟）

### Slide 19: 核心创新点总结（1.5 分钟）

**讲解词**：
总结一下，我们的方案有四个核心创新点：

**创新点 1：Token Passing 中心调度**。

我们放弃了分布式协商（RTS/CTS）和静态时隙分配（TDMA），采用中心化的 Token Passing 机制。服务器完全控制发射权，无 Token 不发射，从根本上消灭了碰撞。

实测结果：**碰撞率 0%**。

**创新点 2：动态水位线与三态状态机**。

我们不是简单地分配固定时隙，而是根据节点的实际需求动态调整窗口。睡眠态、需量拟合态、激进扩窗态，三态状态机自动适应突发流量。

实测结果：**有效带宽利用率 > 90%**。相比静态 TDMA（通常只有 30-40%），这是一个巨大的提升。

**创新点 3：单进程架构与实时调度**。

合并 TUN 读取与发射逻辑，配合 SCHED_FIFO 实时提权，彻底解决了双进程 UDP 通信的数据丢失问题，并将调度抖动控制在微秒级。

**创新点 4：三层分离架构**。

应用层（UFTP）零感知，继续使用标准 UDP/IP；OS 层提供缓冲和反压；物理层完全控制发射时机。各司其职，避免跨层耦合。

这意味着：**UFTP 无需修改任何代码**，就可以享受 Token Passing 的好处。

最终性能指标：
- 碰撞率：**0%**
- 有效带宽利用率：**> 90%**
- 频谱占用（空闲时）：**< 15%**

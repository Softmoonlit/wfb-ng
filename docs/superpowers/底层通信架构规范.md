# 🚀 联邦学习无线空口底层传输架构设计书
**—— 基于深缓冲区与动态时间配额轮询的无碰撞传输架构**

## 一、 系统背景与核心痛点
*   **物理层痛点**：底层的 `wfb-ng` 基于 WiFi Monitor 模式，**没有 CSMA/CA 冲突避免机制**。多节点同时发送会导致灾难性的物理层射频碰撞，丢包率趋近 100%。
*   **业务层痛点**：联邦学习具有**非对称性**（1个服务器对10个客户端）、**突发性**（40MB模型/梯度文件突发传输）和**异步性**（各客户端本地训练时间随机）。
*   **核心解法**：废弃不可控的微观并发，采用**“中心强管控的时间配额轮询（Token Passing）”**机制，结合 Linux 操作系统的**“深水池缓冲队列”**，将碰撞问题在 MAC 层彻底消灭，并对上层业务（UFTP）保持绝对透明。

---

## 二、 三层网络栈架构设计

系统自上而下严格分为三层，各司其职，**严禁跨层耦合**：

### 1. 应用层：UFTP 与 FL 业务逻辑（零感知层）
*   **职责**：负责按序、可靠地组播下发/收集 40MB 的模型文件。
*   **开发要求**：
    *   **无需修改 UFTP 源码**，无需改用 Unix Socket。继续使用标准的 UDP/IP 协议。
    *   **精准的微超发限速（防 Bufferbloat 与断连）**：在 UFTP 启动时使用 `-R` 设置限速，**切忌设置过大**！速率应当设为**单节点期望分摊带宽的 1.1 ~ 1.2 倍**。计算方式为：`限速 = (底层物理速率 / 节点总数) × 1.2`。例如 10 节点均分 10Mbps 物理带宽时设为 1.2Mbps，均分 100Mbps 物理带宽时设为 12Mbps。保留一点微小的超发量以确保物理层时间片到来时有包可发（防饿死），同时防止底层队列无限积压导致 UFTP 状态机超时崩溃。
    *   **适配性超时阈值**：适度调大 UFTP 的 `GRTT`（估计往返时间）和重传尝试次数。配合底层的“浅水池”改动，即使 10 个节点并发轮询，最大物理排队延迟也能控制在数秒内。应用层只需容忍秒级的合理波动即可稳定传输。

### 2. 操作系统层：Linux TUN 虚拟网卡（“极小水池”防抖动）
*   **职责**：作为进程内核交互的“微减震器”，确保存取平滑，并将满载反压迅速传递给应用层。
*   **开发要求**：
    *   初始化 `wfb0`（TUN网卡）时，设置一个**极小的值**（例如 `ifconfig wfb0 txqueuelen 100`）。绝对不能设得过大！排队包过多不仅增加无谓的延迟，还会干扰上层应用的拥塞控制。缓冲的重任必须上浮，交给单进程内的“用户态浅水池”（见下文）。
    *   **原理（丢包自适应反压）**：由于 UFTP 的微超发特性（注水略快于泄洪），业务数据会先填满“用户态浅水池”。当用户态浅水池爆满停止读网卡时，TUN 队列这仅有的 100 个名额也会瞬间填满。一旦内核队列溢出，后续的 UDP 包将直接受到内核的限流（返回 `ENOBUFS` 或被丢弃）。对于 UFTP 而言，这种无长延迟的丢包正是最直接有效的拥塞控制信号（背压），能促使其立刻响应并退避，从而彻底消除 Bufferbloat 导致的深层状态机超时。

### 3. MAC/物理层：魔改 wfb-ng（无碰撞调度层）
*   **职责**：彻底接管物理发射权。没有收到服务器指令前，绝对静默。
*   **开发要求**：不频繁启停 `wfb_tx` 进程，而是在其 C/C++ 主循环中加装**“逻辑阀门”**。

## 三、 核心工作流：空口半双工调度状态机

由于 WiFi 是半双工的，服务器（中心点）必须控制整个空口的收发节奏。服务器端的**调度器（Scheduler）**需无限循环执行以下三个“宏观相位（Phases）”：

### 相位一：下行广播与“呼吸”反馈（Server 发 40MB 模型）
服务器在下发大模型时，不能一气呵成，必须给客户端留出回传 NACK（重传请求）的窗口。
*   **呼吸周期策略（Downlink Breath Cycle，合理且必需）**：相位一的场景是典型的非对称宏观时分复用。Server 是绝对的主导发射方，必须由它主动“停火”，客户端才有机会把积压在本地浅水池中的 NACK 控制包通过射频发射出来。典例周期为 1000ms（可结合模型大小与带宽动态调整）。
    *   **呼出 (800ms)**：服务器的阀门全开，全速向空中注入 40MB 模型包。客户端全员静默接收，产生的 NACK 在各自的浅水池排队。
    *   **吸气 (200ms)**：服务器暂停下发，切为接收态。依次向 10 个客户端预留 5ms 保护间隔后，下发 **NACK回传令牌（NACK_WINDOW，如 25ms）**。
    *   **反馈泄洪与概念澄清**：客户端收到包含序号的令牌后，利用这 25ms 把水池里的 NACK 真实数据包发回服务器。
        > *注：这 `25ms` 绝不是后期微探询（`15ms`）那种空敲门包。UFTP 的 NACK 包虽小，但也包含完整的 IP/UDP 头和前向纠错（FEC）编码载荷。25ms 的宽裕窗口是专门预留给客户端将真实的“重传请求数据”稳妥打入空中的开火时间（包含硬件发射耗时）。*
*   *此相位循环执行，直至 40MB 下行全部完成且 UFTP 确认 DONE。*

### 相位二：分散交织探询与自适应弹性轮询泄洪（AEW）
下行结束后，客户端进入漫长的本地 FL 训练。由于各节点训练完成的时间极度不确定且存在高度非对称性（例如 1个活跃传参，9个仍在深睡），系统必须废弃传统的“集中点名风暴”与“固定周期探询”，采用**活跃队列（AQ）与睡眠队列（SQ）的动态交织调度**，彻底平滑射频信道占用。

*   **机制核心：双队列调度**
    *   **活跃队列 (AQ - Active Queue)**：当前正在回传数据（如 40MB 模型参数）的节点。
    *   **睡眠队列 (SQ - Sleep Queue)**：当前处于本地训练状态，或暂时无数据发送的节点。

#### 场景 1：全员深睡探询（Idle Patrol Mode）
当所有客户端都在训练，AQ 队列为空时：
*   **动作（消除探询风暴）**：服务器进入“空闲巡逻模式”。每次**仅从 SQ 队列队首抽出 1 个节点**，向其下发 15ms 的微探询 Token（放宽至 15ms 以覆盖非实时系统的调度毛刺，避免假死误判）。如果节点无数据响应，则将其放回 SQ 队尾。
*   **平滑打散（IDLE_PATROL_INTERVAL）**：探询完一个节点后，服务器强制休眠（或静默侦听）一段较长的基础时间（如 `100ms`）。对于动辄耗时几分钟的联邦学习训练而言，晚唤醒 1-2 秒对全局进度毫无影响，完全可以用微小的唤醒延迟换取极低的频谱底噪。
*   *优势*：针对 10 个节点，原本 50ms 集中并发打完的探询风暴，被均匀打散到 1150ms（10次 * (15ms+100ms间隔)）的时间轴上。此时无线信道的频谱占用率极低（约 13%），没有任何冲击峰值，且极度省电。

#### 场景 2：混合交织调度（Mixed Interleaved Mode）
当部分节点完成训练进入 AQ 队列开始“狂飙”发数据，而其他节点仍在 SQ 队列中训练时：
*   **动作（按比穿插探询，保护活跃吞吐）**：服务器必须保证 AQ 队列享有绝对的主导带宽。为了避免连续探询多个 SQ 节点（比如连续 9个节点耗时 135ms）导致 AQ 节点的 UFTP 传输产生剧烈的网络抖动，引入**交织探询比（Interleave Ratio, 例如 N=3 到 5）**。
*   *执行逻辑*：
    *   服务器连续服务 AQ 队列 $N$ 次（向活跃节点发放 50~300ms 的弹性大口径 Token）。
    *   **节点切换保护**：在给不同节点发放 Token 之间，必须强制插入 **5ms 的保护间隔（Guard Interval）**，以确保上一节点硬件 TX Queue 中的残余射频包彻底排空，避免空中撞车。
    *   每服务 $N$ 次 AQ 后，**仅抽出 1 个** SQ 队列的节点发送 15ms 微探询。探询完毕后立刻回到 AQ 服务。
*   *优势*（以 1 活跃 9 睡眠为例，设 N=4）：活跃节点连续拿到 4 次大窗口（累计近 1200ms 的满载带宽），期间只被打断 1 次 15ms 去微探询睡眠节点 1（探询开销仅占 1.2%）。这 15ms 对于活跃节点完全无感，而随着时间推移（约 10.8 秒一轮），9个睡眠节点会在后台被悄无声息地轮询完毕。

#### 场景 3：状态跃迁与自适应弹性扩窗（AEW）
不论是 AQ 还是 SQ 中的节点，其每次获得的 Token 授权时间（`duration_ms`）均受【三态状态机引擎】控制，独立动态伸缩：

*   **核心参数定义（经排队论推演）**：
    *   `MIN_WINDOW` = **50 ms**（刚需底线，主要用于覆盖 OS 调度抖动约 20ms 及零负载侦听底线 30ms）。
    *   `MAX_WINDOW` = **300 ms**（针对 10 节点并发）或 **600 ms**（针对 5 节点以内并发）。推演规则：必须保证 `(N-1) * MAX_WINDOW` 严苛小于业务层 UFTP 等协议的断连超时上限（GRTT）。
    *   `STEP_UP` = **100 ms**（进攻扩窗步长，能在 2~3 个周期快速将占空比拉满，切合大模型突发传输）。
    *   `ELASTIC_MARGIN` = **50 ms**（弹性宽容余量，包容操作系统的任务调度毛刺与 UFTP 投递抖动）。

*   **【三态状态机与跃迁引擎】动态授权规则**：
    每次 Server 收集到节点本次实际发射数据的耗时（`actual_used_time`，不含测定静默的 5ms 空等损耗）后，执行以下流水线计算其下一轮的时隙授权及队列归属：
    1.  **状态 I：极速枯竭与睡眠 (Zero-Payload Drop) & Watchdog 防误判保护**
        *条件*：`actual_used_time < 5ms`（几乎无有效包，极速早停）。
        *动作*：引入 **Watchdog 动量拦截**。如果该节点上一轮处于状态 III（高负载），此次突然绝对静默，大概率是 Token 在空中受干扰丢失。此时**保持原有时隙大小，不自增沉默计数器，并在下一轮优先重发 Token**。只有当**连续 3 次**探询均绝对静默时，才真正判定为断流，`next_window = MIN_WINDOW`，并将其**从 AQ 踢出，放入 SQ 队尾**。
    2.  **状态 II：唤醒、需量拟合与退避 (Demand-Fitting)**
        *条件*：`5ms <= actual_used_time < allocated_window`（有效发放，但中途断档触发 5ms 物理静默的 Early-Abort）。
        *动作*：若该节点在 SQ 中，**立刻将其拔出并插入 AQ 队尾**（完成唤醒）。`next_window = max(MIN_WINDOW, actual_used_time + ELASTIC_MARGIN)`。**绝对不乘性减半！** 紧贴消耗量并加上弹性余量，保留下一轮爆发的可能，同时清零“连续沉默计数器”。
    3.  **状态 III：高负载贪突发 (Additive Increase)**
        *条件*：`actual_used_time >= allocated_window`（完全用光了分配的时隙，未触发 Early-Abort）。
        *动作*：必在 AQ 中，`next_window = min(MAX_WINDOW, allocated_window + STEP_UP)`。说明积压严重，果断加法扩窗，加速大模型满速卸载。

## 四、 核心代码修改指南（Developer Checklist）

请开发团队重点关注以下三个模块的编码工作：

### 1. 定义超轻量级控制帧 (MAC Control Frame) & 数据面控制面分离
在 `wfb-ng` L2 物理帧注包前，直接在 Radiotap Header 后携带极简 Token，**脱离 FEC 与 IP 路由体系**。
```c
struct TokenFrame {
    uint8_t  magic;       // 魔数，用于快速区分控制帧与数据帧 (如 0x02)
    uint8_t  target_node; // 目标节点编号 (0=广播/ALL)
    uint16_t duration_ms; // 授权发射时间片 (毫秒)
    uint32_t seq_num;     // 【关键】全局单调递增序号（Server在整个生命周期统一生成维护），防止 RX 缓冲区旧包诈尸
};
```
**【发送端(Server)特别规范】：**
1. **跳过 FEC**：Token 严禁进行 FEC 纠错编码，避免积压组块延迟！
2. **单包独立降维 (双 Radiotap 模板分流)**：这是最合理且最高效的实现方式。在 C/C++ 代码启动初始化时，**生成并保存两份 Radiotap 模板**：一份从 `wfb_tx` 启动参数解析而来（用于发高阶 MCS 业务数据），另一份在代码中硬编码锁定为 **最低 MCS 速率（如 MCS 0 或 1Mbps）**（用于发 Token 控制帧）。在底层发射的主循环中，通过判断帧类型实现**分流发射**，分别拼接对应的 Header。这样既利用了 Monitor 模式单包定速的天然特性确保 Token 全网覆盖，又避免了每次发包重新构建 Header 的 CPU 开销，是工程上最佳的设计。
3. **三连发击穿**：因为缺乏 FEC，发 Token 时在一个 C 循环里连续无脑调用 `pcap_inject()` 3 遍，接收端靠 `seq_num` 自动去重。
4. **全局甜点发射功率 (非代码层面，运维规范)**：切勿试图在 C++ 代码中通过 Radiotap 单包修改 TX Power（RTL8812AU 等驱动大概率会静默抹杀）。应在节点启动脚本中，直接设置全局“甜点”功率（如 `iwconfig wlan0 txpower 24`，约 250mW）。此功率下能完美兼顾数据帧的高阶调制波形失真（EVM 容忍度）和 Token 帧的顶尖穿透力，且避开网卡过热与电流断崖死机的硬件雷区。

### 2. 客户端与服务器底层网卡的“架构整合”（物理内存分配与逻辑动态背压）
**【关键陷阱警告】**：“深水池”失效导致的数据静默丢弃。原版 wfb-ng 架构中，网卡读取进程(`wfb_tun`) 和空口发射进程(`wfb_tx`) 是独立的，两者通过无阻塞的本地 **UDP Socket (127.0.0.1)** 通信。当 `wfb_tx` 逻辑阀门关闭（如吸气期停火，或等待 Token）时，`wfb_tun` 仍会疯狂抽干 TUN 队列并塞向 UDP。一旦 UDP 接收缓冲区（`rmem`）打满，内核会直接丢弃后续包（`ENOBUFS`）。**结果是数据根本憋不住，直接在内部通信环路丢失。**

**开发要求（单进程化与动态双层缓冲架构）**：禁止使用两个独立进程通过 UDP 通信。必须将 `wfb_tun` 的读取逻辑与 `wfb_tx` 的发射逻辑**合并到同一个进程中**。

内部的“用户态浅水池”必须进行**物理分配与逻辑水位线的彻底解耦**：
1. **物理内存池 (Physical Buffer)**：在进程启动时，静态分配一个绝对安全的环形队列（例如 `std::vector`，容量 10000 包，约 15MB），彻底杜绝 `bad_alloc` 内存溢出崩溃。
2. **逻辑反压水位线 (Logical Watermark)**：这是控制 `tun_reader` 停手不读的核心原子变量（`dynamic_queue_limit`）。它**绝对不能硬编码**，必须基于【当前底层物理速率 $R_{phy}$】和【憋气/开火时间】动态计算。
   *   **MCS 动态映射**：C++ 代码必须解析启动参数传入的 MCS（调制编码策略）和频宽（20/40MHz），通过内置查找表映射出实时的理论物理速率 $R_{phy}(bps)$（例如 MCS 0 对应 6Mbps，MCS 4 对应 135Mbps）。若硬编码为 100Mbps 却跑在 6Mbps 下，算出的几千个包在空口需要数秒才能排空，会引发灾难性的 Bufferbloat 和 UFTP 断连！
   *   **Server 端公式（防吸气憋死）**：当在线节点数 $N$ 变化时，重新计算水位线，刚好兜住吸气期注入的数据。
       $\text{Server\_Limit} = \lceil \frac{R_{phy}(bps) \times \text{Inhale\_Time}(s) \times 1.5}{8 \times MTU} \rceil$，其中 $\text{Inhale\_Time} = N \times (\text{NACK\_WINDOW} + \text{GUARD\_INTERVAL})$。
   *   **Client 端公式（防开火断流）**：收到 Token 时，依据本次授权窗口重新计算水位线，确保弹药充足。
       $\text{Client\_Limit} = \lceil \frac{R_{phy}(bps) \times \text{duration\_ms} \times 1.5}{8 \times MTU \times 1000} \rceil$。

当浅水池的积压包数达到动态水位线时，`tun_reader` 线程立刻休眠停止调用 `read()`，让压力倒灌回操作系统的 TUN 队列（需设置极小的 `txqueuelen 100`），从而引发底层队列溢出丢包，向应用层 UFTP 返回 `ENOBUFS` 触发拥塞退避。

```c
#include <atomic>
#include <poll.h>
#include <condition_variable>
#include <mutex>
#include <sched.h>
#include <pthread.h>

std::atomic<bool>     can_send{false};
std::atomic<uint64_t> token_expire_time_ms{0};
static uint32_t       max_seen_seq = 0;

// 消除 usleep 延迟的终极武器：内存级零延时条件变量
std::condition_variable cv_send;
std::mutex              cv_m;

// 【全局动态逻辑水位线】
std::atomic<size_t>   dynamic_queue_limit{3000}; 

// 提权工具函数，用于抵御非实时 Linux 内核的 CFS 调度 Jitter (抖动)
void set_thread_realtime_priority(int priority) {
    pthread_t this_thread = pthread_self();
    struct sched_param params;
    params.sched_priority = priority;
    if (pthread_setschedparam(this_thread, SCHED_FIFO, &params) != 0) {
        printf("Warning: Must run as root to set SCHED_FIFO for 5ms precise timing!\n");
    }
}

// 【控制面/数据面硬分流层】：高速路由分发与抽干队列机制
void* rx_demux_thread() {
    set_thread_realtime_priority(99); // 控制面（收包）最高实时优先级
    
    struct pollfd fds[1];
    // 假定 raw_sock_fd 为监听网卡的 RAW Socket fd
    fds[0].fd = raw_sock_fd; 
    fds[0].events = POLLIN;

    while(1) {
        // 使用 poll 替代 usleep，实现内核级硬件中断极速唤醒，消除 1ms 级别的无意义忙等损耗
        int ret = poll(fds, 1, -1); 
        if (ret <= 0) continue;

        TokenFrame latest_token;
        bool got_new_token = false;

        // 1. 【极速抽干队列】：用非阻塞模式一次性把网卡底层的积压包全拿出来
        while (recv_from_air_nonblocking(&raw_pkt) > 0) { 
            if (is_control_frame(raw_pkt)) {
                TokenFrame token = parse_token(raw_pkt);
                if (token.seq_num > max_seen_seq) { // 序号校验，拦截过期历史 Token
                    max_seen_seq = token.seq_num;
                    latest_token = token;
                    got_new_token = true; 
                }
            } else if (is_data_frame(raw_pkt)) {
                // 业务数据包（包含 UFTP 的 ACK），走正常 FEC 解码后塞入 TUN
                FecPacket fec_pkt = decode_fec(raw_pkt);
                if (fec_is_valid(fec_pkt)) {
                    write_to_tun(wfb_tun_fd, fec_pkt.ip_payload);
                }
            }
        }

        // 2. Client：根据本批次【最新】的合法令牌动态收缩/扩张水位线
        if (got_new_token) {
            if (latest_token.target_node == MY_NODE_ID) {
                token_expire_time_ms.store(get_current_time_ms() + latest_token.duration_ms);
                
                // 唤醒发射线程（零轮询延迟）
                {
                    std::lock_guard<std::mutex> lk(cv_m);
                    can_send.store(true);
                }
                cv_send.notify_one(); 
                
                // 实时物理速率从 MCS 映射表获取，确保水位线随信道质量动态拟合
                size_t new_limit = calculate_dynamic_limit(latest_token.duration_ms, current_mcs_phy_rate_bps);
                dynamic_queue_limit.store(new_limit, std::memory_order_relaxed);
            } else {
                can_send.store(false); 
            }
        }
    }
}

// 【系统架构合并优化】：把原 wfb_tun (读取) 和 wfb_tx (发射) 的逻辑重构到一起
// 物理内存池静态分配 10000 包容量（约 15MB），彻底杜绝内存溢出崩溃
ThreadSafeQueue<Packet> internal_queue(10000); 

// [独立子线程 A：读源头（装弹手） - 负责读取、FEC预处理与制造反压]
void tun_reader_thread() {
    // 【致命陷阱防范】：切勿将此线程设为 SCHED_FIFO。FEC 编码是纯 CPU 算力怪兽，
    // 设为实时优先级会引发全系统死机。保留在普通 CFS 调度队列，依靠 1.5 倍底仓容忍其调度延迟。
    
    while(1) {
        // 实时读取当前的动态逻辑水位线
        size_t current_limit = dynamic_queue_limit.load(std::memory_order_relaxed);

        // 如果内部队列积压达到逻辑水位，立刻停住不读网卡！
        // 核心意义：将“无包可发”转化成物理反压，迫使大数据一直憋在操作系统级的 queue 中
        if (internal_queue.size() >= current_limit) {
            // 此处用 usleep 没问题，因为底仓丰厚且只是为了退避让出 CPU
            usleep(1000); 
            continue;
        }
        
        Packet pkt = read_from_tun(); // 从 /dev/net/tun 获取原始 IP 流量
        FecPacket fec_pkt = encode_fec(pkt); 
        internal_queue.push(fec_pkt); 
        
        // 刚装了一发弹，尝试唤醒可能因为缺弹而沉睡的射手
        cv_send.notify_one(); 
    }
}

// [独立子线程 B：发射端（射手） - 空口阀门总控，纯物理发射]
void tx_main_loop() {
    set_thread_realtime_priority(90); // 射频发射（射手）次高实时优先级
    
    while (1) {
        // 使用条件变量实现极速唤醒，彻底废弃了会带来 1ms~5ms 灾难性抖动的 usleep() 盲等
        std::unique_lock<std::mutex> lk(cv_m);
        cv_send.wait(lk, []{ 
            return (can_send.load() && !internal_queue.empty()) || 
                   (get_current_time_ms() > token_expire_time_ms.load()); 
        });
        lk.unlock(); // 立刻释放锁，准备开火

        // 逻辑原子阀门：没拿到令牌或者令牌过期，绝对不开火
        if (!can_send.load() || get_current_time_ms() > token_expire_time_ms.load()) {
            can_send.store(false);
            continue;
        }

        // pcap_inject()：直接将成品射频包打入空中！不占用时隙进行 CPU 计算
        FecPacket fec_pkt = internal_queue.pop();
        pcap_inject(fec_pkt); 
    }
}
```

### 3. 服务器调度守护进程 (Server Scheduler)
**强制使用纯 C/C++ 编写**（严禁使用 Python 等带 GC 的语言，以彻底消除运行时抖动毛刺）。必须配合 **RAW Socket / libpcap (Monitor 模式)** 绕过内核网络栈，直接在物理层注入控制帧以确保极致低延迟。
*   **无状态的 `Active_Nodes` 动态列表**：结合控制信道极高可靠性（最低 MCS 降维 + 三连发击穿）与 `Micro_Polling` 机制，实现极速的心跳检测。节点超时掉线即踢出并降级为探询对象，一旦收到探询回应瞬间重新挂载，**严禁引入任何阻塞式的“状态同步等待”或“超时重传”，从架构上彻底杜绝死锁**。
*   维护当前的 `System_Phase` (Downlink_Broadcast, Micro_Polling, Uplink_Burst)。
*   **硬核高精度定时与实时提权**：内置全局系统级的 `seq_num` 计数器防旧包诈尸。鉴于 Linux 用户态非实时特性，调度器主线程**必须提权至 `SCHED_FIFO` (最高实时优先级)**，且强烈建议抛弃精度极差的 `usleep()`，改用 `timerfd` 配合截止前极短时间的空转自旋（Spin-wait），确保按状态机以**微秒级绝对精度**周期性向网卡注入 `TokenFrame`。

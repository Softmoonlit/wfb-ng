### Slide 4: 射频碰撞问题演示

**核心要点**:
- **问题本质**: 多节点同时发送 → 物理层射频碰撞 → 数据丢失
- **实测数据**: 碰撞率可达 30-50%，严重影响传输效率
- **连锁反应**:
  - 应用层超时重传 → 更大拥塞
  - 有效带宽利用率降至 20% 以下
  - 联邦学习训练中断

**图表说明**:

### Slide 7: Token Passing 核心思想

**核心要点**:
- **类比**: 交通信号灯控制路口
  - 只有获得"绿灯（Token）"的节点才能发送
  - 其他节点必须等待

- **核心机制**:
  - 中心调度器（服务器）控制发射权
  - 按需分配时间配额（动态窗口）
  - 无 Token 不发射，彻底消灭碰撞

**图表说明**:
- 交通信号灯示意图：红灯停，绿灯行
- Token Passing 概念图：服务器发放 Token，节点收到 Token 才能发送

**讲解要点**:
- 用交通信号灯类比，非常直观
- 强调"中心化控制"的必要性

### Slide 8: 三层架构设计（全局视图）

**核心要点**:
- **应用层（UFTP）**:
  - 职责：按序、可靠地组播下发/收集 40MB 模型文件
  - 零感知：无需修改 UFTP 源码，使用标准 UDP/IP

- **OS 层（TUN + 调度）**:
  - 职责：深缓冲区（环形队列），动态背压
  - 作用：平滑突发流量，防止 Bufferbloat

- **物理层（魔改 wfb-ng）**:
  - 职责：Token Passing 调度，彻底接管物理发射权
  - 无 Token 不发射

**图表说明**:
- 三层架构图（纵向分层）：
  - 顶层：应用层（UFTP）
  - 中层：OS 层（TUN 设备、环形队列、调度器）
  - 底层：物理层（Monitor 模式、Token Passing）

**讲解要点**:
- 强调"各司其职，避免跨层耦合"
- 标注数据流向：应用层 → OS 层 → 物理层

### Slide 9: 为什么选择三层分离？

**核心要点**:
- **原因 1**: 应用层无需修改 UFTP 源码
  - UFTP 使用标准 UDP/IP，对底层调度完全透明

- **原因 2**: OS 层提供缓冲和反压，防止 Bufferbloat
  - 极小 TUN 池（txqueuelen=100）+ 用户态深缓冲
  - 快速丢包反压，避免应用层超时假死

- **原因 3**: 物理层完全控制发射时机
  - Token Passing 调度，无碰撞

- **设计原则**: 各司其职，避免跨层耦合

**图表说明**:
- 对比图：三层分离 vs 单层紧耦合
- 标注"解耦"的好处

**讲解要点**:
- 解释"为什么要分层"，而不是全部在物理层实现
- 强调"应用层零感知"的核心价值

### Slide 10: Token Passing 的工作流程

**核心要点**:
- **步骤 1**: 服务器广播 Token（控制帧）
  - 指定目标节点 ID 和时间窗口大小

- **步骤 2**: 指定节点获得发射权
  - 解析 Token，确认是自己

- **步骤 3**: 节点在时间窗口内发送数据
  - 从队列中取出数据包，通过物理层发射

- **步骤 4**: 窗口到期，服务器切换到下一个节点
  - 插入 5ms 保护间隔，避免空中撞车

**图表说明**:
- Token Passing 流程图（时序图）：

### Slide 11: 动态时间配额分配

**核心要点**:
- **不是固定时隙**: 根据节点实际需求动态调整窗口大小
- **三态状态机**:
  - 睡眠态：节点长时间静默，窗口降为 0
  - 需量拟合态：按实际发送量分配
  - 激进扩窗态：节点有大量数据，激进扩大窗口

- **关键参数**:
  - MIN_WINDOW: 50ms（容忍 OS 调度抖动）
  - MAX_WINDOW: 300ms（避免其他节点饥饿）

**图表说明**:

### Slide 13: 应用层数据注入 → TUN 极小水池

**核心要点**:
- **流程**: UFTP 发送 40MB 模型数据，通过标准 UDP/IP 到达 TUN 设备

- **关键技术 1: TUN 极小水池**
  - txqueuelen = 100（不是默认的 1000）
  - 为什么？触发快速丢包反压，防止 Bufferbloat
  - 效果：向 UFTP 返回 ENOBUFS 拥塞信号，避免应用层超时假死

**图表说明**:
- TUN 设备队列示意图：
  - 绘制一个小水池（容量 100），标注"极小水池"
  - 水池满了 → 丢弃数据包 → 返回 ENOBUFS

**讲解要点**:
- 用"水池"类比：水池太小，水满了就溢出（丢弃），防止下游堵塞
- 强调"快速丢包反压"的重要性

### Slide 14: TUN 读取 → 环形队列深缓冲

**核心要点**:
- **流程**: 控制面线程从 TUN 读取数据包

- **关键技术 2: 线程安全环形队列**
  - 静态分配 10000 包容量，无动态内存分配
  - 队列满时直接丢弃，防止内存溢出
  - 线程安全：使用互斥锁保护

- **关键技术 3: 动态水位线机制**
  - 问题：如何判断"队列多少算满"？
  - 计算：基于物理速率动态计算
    ```
    limit = (PHY_Rate × duration / MTU) × 1.5
    ```
  - 达到水位线 → 停止读取 TUN，产生背压

**图表说明**:
- 环形队列示意图：
  - 绘制一个环形缓冲区，标注"容量 10000"
  - 标注"水位线"（动态计算）
  - 队列满 → 丢弃数据包

**讲解要点**:
- 解释"为什么要水位线"：不是等队列真的满了才停止，而是提前预警
- 强调"动态"的重要性：不同 MCS 速率对应不同水位线

### Slide 15: 调度器决策 → 三态状态机调度

**核心要点**:
- **流程**: 服务端调度器主线程（SCHED_FIFO 优先级 99）运行

- **关键技术 4: 三态状态机调度**
  - 问题：如何动态分配时间窗口？
  - 状态 I（睡眠）：节点连续 3 次静默 → 窗口降为 0
  - 状态 II（需量拟合）：`window = max(MIN, actual + MARGIN)`
  - 状态 III（激进扩窗）：`window = min(MAX, allocated + STEP)`
  - 为什么需要三态？适应突发流量，平衡效率与公平

- **关键技术 5: AQ/SQ 双队列管理**
  - AQ（活跃队列）：有数据要发的节点
  - SQ（睡眠队列）：长时间静默的节点
  - 混合交织：每 N 次 AQ 服务穿插 1 次 SQ 探询

**图表说明**:
- 三态状态机完整跃迁图：
  - 状态 I ↔ 状态 II ↔ 状态 III
  - 标注跃迁条件（如"连续 3 次静默"）
- AQ/SQ 队列示意图：
  - 绘制两个队列，标注节点状态迁移

**讲解要点**:
- 解释"为什么是三态，而不是两态或四态"：平衡复杂度和灵活性
- 强调"混合交织"的作用：避免睡眠节点饿死

### Slide 16: Token 生成 → 授权控制帧

**核心要点**:
- **流程**: 调度器决定下一个发射节点，生成 TokenFrame

- **关键技术 6: MAC 控制帧设计**
  - 结构：
    ```
    TokenFrame {
      uint8_t magic;         // 魔数，标识帧类型
      uint8_t target_node;   // 目标节点 ID
      uint16_t window_size;  // 时间窗口大小（ms）
      uint32_t seq_num;      // 序列号，防止重放
    }
    ```
  - 序列化：固定结构体直接内存拷贝（零拷贝）
  - 全球单调递增序列号，防止重放攻击

- **关键技术 7: 控制帧跳过 FEC**
  - 为什么？FEC 编码会引入 3-5ms 延迟积压
  - 控制帧很小（< 100 字节），不需要分片和纠错

**图表说明**:
- TokenFrame 结构图：
  - 绘制结构体字段，标注每个字段的字节数
  - 总大小：8 字节（紧凑对齐）

**讲解要点**:
- 解释"为什么要序列化"：网络传输需要二进制格式
- 强调"跳过 FEC"的重要性：降低延迟

### Slide 17: 发射端线程唤醒 → Radiotap 模板选择

**核心要点**:
- **流程**: 发射端线程通过 condition_variable 零轮询唤醒

- **关键技术 8: 零轮询唤醒机制**
  - 问题：usleep() 会导致毫秒级延迟毛刺
  - 方案：使用 std::condition_variable，精准唤醒
  - 效果：微秒级响应

- **关键技术 9: Radiotap 模板分流**
  - 控制帧 → 使用最低 MCS 速率（6Mbps）
    - 为什么？确保全网覆盖，即使远距离节点也能接收
  - 数据帧 → 使用高阶 MCS（如 MCS 8，144Mbps）
    - 为什么？最大化吞吐量
  - 发射时零延迟切换，提前准备两套模板

**图表说明**:
- Radiotap 模板选择流程图：
  ```
  判断帧类型 → 控制帧 → 使用 MCS 0 模板
             → 数据帧 → 使用 MCS 8 模板
  ```
- 对比图：MCS 0（6Mbps，覆盖远）vs MCS 8（144Mbps，覆盖近）

**讲解要点**:
- 解释"Radiotap 是什么"：WiFi 物理层控制头，决定发射速率
- 强调"零延迟切换"的重要性：提前准备，不等待

### Slide 18: 空口发射 → 三连发击穿与保护间隔

**核心要点**:
- **关键技术 10: 三连发击穿机制**
  - 问题：无线环境可能丢包，控制帧丢失会怎样？
  - 方案：每个控制帧连续发送 3 次
  - 接收端通过序列号去重
  - 效果：大幅提高控制帧可靠性

- **关键技术 11: 节点切换保护间隔**
  - 问题：节点 A 刚发完，节点 B 立即发送，可能空中撞车
  - 方案：服务器在发送下一个 Token 前插入 5ms 保护间隔
  - 为什么是 5ms？覆盖硬件延迟和传播延迟

**图表说明**:
- 节点切换时序图：
  ```
  节点 1 发送数据 → [5ms 保护间隔] → 服务器发送 Token(节点 2) → 节点 2 发送数据
  ```
- 三连发示意图：
  ```
  Token(seq=1001) → Token(seq=1001) → Token(seq=1001)
  ```

**讲解要点**:
- 解释"为什么是三连发，不是两连发或四连发"：平衡可靠性和开销
- 强调"保护间隔"的重要性：半双工切换需要时间

### Slide 19: 接收端处理 → 状态反馈与窗口调整

**核心要点**:
- **流程**:
  - 客户端收到 Token，解析 target_node 和 window_size
  - 如果是自己 → 在窗口内发送数据
  - 如果不是自己 → 保持静默，等待自己的 Token
  - 发送完毕后，服务器根据实际发送量调整窗口

- **关键技术 12: 空闲巡逻模式**
  - 问题：如果所有节点都静默（队列空），如何发现新数据？
  - 方案：每 100ms 探询一个节点（轮询 SQ 队列）
  - 效果：降低频谱占用，避免固定周期探询风暴
  - 频谱占用：< 15%

**图表说明**:
- 接收端状态机：
  ```
  等待 Token → 判断是否自己 → 是：发送数据 → 返回等待
                            → 否：继续等待
  ```
- 空闲巡逻时序图：
  ```
  [100ms] → 探询节点 1 → 无数据 → [100ms] → 探询节点 2 → 有数据 → 切换到活跃状态
  ```

**讲解要点**:
- 解释"空闲巡逻"的作用：在空闲时保持低频谱占用
- 强调"混合交织"：活跃节点和睡眠节点都要照顾

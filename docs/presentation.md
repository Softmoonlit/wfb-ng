# 联邦学习无线空口底层传输架构设计

## 基于 Token Passing 的无碰撞传输机制

汇报人：陈龙滨
日期：2026-04-25

---

# 第一部分：问题背景与挑战

---

## Slide 2: 联邦学习场景与挑战

### 场景描述
- **1 个服务器 + 10 个客户端**，周期性传输 40MB 模型参数

### 三大特征
1. **非对称性**：1 对 N 的星型拓扑
2. **突发性**：大模型文件集中传输
3. **异步性**：各客户端本地训练时间随机，完成时刻不可预测

---

## Slide 3: 无线空口的特殊性

### 半双工通信
- 同一时刻只能发送或接收，不能同时进行

### WiFi Monitor 模式
- 绕过内核网络栈，直接访问物理层
- **无 CSMA/CA 冲突避免机制**（关键问题）
- 类比：没有交通信号灯的十字路口

<!-- 图表占位：半双工通信示意图，对比有信号灯 vs 无信号灯的路口 -->

---

## Slide 4: 射频碰撞问题演示

### 问题本质
多节点同时发送 → 物理层射频碰撞 → 数据丢失

### 实测数据
- 碰撞率可达 **30-50%**，严重影响传输效率

### 连锁反应
- 应用层超时重传 → 更大拥塞
- 有效带宽利用率降至 **20% 以下**
- 联邦学习训练中断

<!-- 图表占位：碰撞示意图，多车同时过无信号灯路口，撞车 -->

---

## Slide 5: 现有方案的局限性

### 方案 1: 分布式协商（RTS/CTS）
- **问题**：每次发送前都要握手，开销大
- **效率**：有效吞吐量 < 50%

### 方案 2: 静态 TDMA 时隙分配
- **问题**：无法适应突发流量，空闲时隙浪费
- **场景不适配**：联邦学习的异步性导致大部分时隙空闲

### 方案 3: 应用层限速
- **问题**：无法根治物理层碰撞，只是缓解

---

## Slide 6: 核心挑战总结

### 挑战 1: 如何在物理层彻底消灭碰撞？
→ 需要中心化的发射权控制

### 挑战 2: 如何对应用层（UFTP）保持透明？
→ 不能修改 UFTP 源码，使用标准 UDP/IP

### 挑战 3: 如何适应联邦学习的突发流量特性？
→ 需要动态时间配额分配，而非固定时隙

**解决方案：Token Passing 中心调度**

# 第二部分：解决方案核心思想

---

## Slide 7: Token Passing 核心思想

### 中心调度机制
- **服务器统一管理发射权**：作为中心调度器控制所有节点
- **轮流授权发射**：每个时刻只授权一个节点发送数据
- **动态时间窗口**：授权时长根据节点数据量动态调整

### 工作原理
1. 服务器生成 Token（令牌），指定目标节点和时间窗口
2. 目标节点收到 Token 后，在窗口内发送数据
3. 窗口到期后，服务器切换到下一个节点
4. **任何时刻空中只有一个节点发射**，彻底消灭碰撞

<!-- 图表占位：Token Passing 时序图，服务器轮流授权节点 -->

---

## Slide 8: 三层架构设计（全局视图）

### 应用层（UFTP）
- 职责：按序、可靠地组播下发/收集 40MB 模型文件
- 零感知：无需修改 UFTP 源码，使用标准 UDP/IP

### OS 层（TUN + 环形队列）
- **服务器/客户端共有**：TUN 虚拟网卡，拦截 UFTP 数据包
- **深缓冲区**：线程安全环形队列（10000 包容量）
- **动态背压**：水位线机制，触发快速丢包反馈

### 物理层（魔改 wfb-ng）
- **服务器特有**：Token Passing 调度器，中心化发射权控制
- **客户端特有**：Token 解析与发射控制，无 Token 不发射
- **通用**：Monitor 模式帧注入，Radiotap 模板分流

# 第三部分：Token Passing 流程与关键技术

---

## Slide 9: Token Passing 流程全景图

### 完整流程
```
┌─────────────────────────────────────────────────────────────────┐
│ 客户端：UFTP → TUN → 环形队列 → 等待 Token → 在窗口内发送 → 空口 │
│                              ↑                                  │
│ 服务器：调度决策 → 生成 Token ──────┘                            │
│         ↑                                                       │
│ 客户端：发送数据 → 服务器接收 → 确认/更新状态 → 发送下一个 Token │
└─────────────────────────────────────────────────────────────────┘
```

### 说明
- 服务器下发模型采用独立的"呼吸周期"策略，本部分聚焦 Token Passing
- 流程涉及**服务器和客户端双方**，共 **14 个关键技术**

<!-- 图表占位：Token Passing 流程图，标注 14 个关键技术点 -->

---

## Slide 10: 应用层数据注入 → TUN 极小水池【客户端】

### 流程
UFTP 发送 40MB 模型数据，通过标准 UDP/IP 到达 TUN 设备

### 关键技术 1: TUN 极小水池
- **txqueuelen = 100**（不是默认的 1000）
- **为什么？** 触发快速丢包反压，防止 Bufferbloat
- **效果**：向 UFTP 返回 ENOBUFS 拥塞信号

<!-- 图表占位：TUN 设备队列示意图，标注"极小水池" -->

---

## Slide 11: TUN 读取 → 环形队列深缓冲【客户端】

### 关键技术 2: 线程安全环形队列
- 静态分配 **10000 包容量**，无动态内存分配
- 达到水位线时停止读取，产生反压（而非丢弃数据包）
- 线程安全：使用互斥锁保护

### 关键技术 3: 动态水位线机制
- **问题**：如何判断"队列多少算满"？
- **计算**：
  ```
  limit = (PHY_Rate × duration / MTU) × 1.5
  ```
- **效果**：达到水位线 → 停止读取 TUN，产生背压

<!-- 图表占位：环形队列示意图，标注水位线 -->

---

## Slide 12: 调度器决策 → 实时调度与三态状态机【服务器】

### 关键技术 4: 实时调度提权
- **问题**：Linux CFS 调度器有 20ms 级抖动
- **解决**：关键线程提权到 SCHED_FIFO
  - 控制面线程：优先级 **99**（最高）
  - 发射端线程：优先级 **90**
  - TUN 读取线程：**普通优先级**（FEC 是 CPU 密集型，防死锁）

### 关键技术 5: 三态状态机调度
- **状态 I（睡眠）**：节点连续 3 次静默 → 迁移到 SQ，探询时使用 15ms 微窗口
- **状态 II（需量拟合）**：`window = max(MIN, actual + MARGIN)`
- **状态 III（激进扩窗）**：`window = min(MAX, allocated + STEP)`
- **Watchdog 保护**：单次静默保持原窗口，连续 3 次才降级

---

## Slide 13: AQ/SQ 双队列管理【服务器】

### 关键技术 6: AQ/SQ 双队列管理

**队列定义**：
- **AQ（活跃队列）**：有数据要发的节点
- **SQ（睡眠队列）**：长时间静默的节点

**混合交织调度**：
- 每 4 次 AQ 服务穿插 1 次 SQ 微探询（15ms）
- 探询间隔 100ms，频谱占用 < 15%

<!-- 图表占位：三态状态机完整跃迁图，AQ/SQ 队列示意图 -->

---

## Slide 14: Token 生成 → 授权控制帧【服务器】

### 关键技术 7: MAC 控制帧设计

**TokenFrame 结构**：
```c
struct TokenFrame {
  uint8_t magic;         // 魔数，标识帧类型
  uint8_t target_node;   // 目标节点 ID
  uint16_t window_size;  // 时间窗口大小（ms）
  uint32_t seq_num;      // 序列号，防止重放
};
```
- 序列化：固定结构体直接内存拷贝（零拷贝）
- 全球单调递增序列号，防止重放攻击

### 关键技术 8: 控制帧跳过 FEC
- **为什么？** FEC 编码会引入 3-5ms 延迟积压
- 控制帧很小（< 100 字节），不需要分片和纠错

---

## Slide 15: 空口发射 → Radiotap 模板分流【服务器】

### 关键技术 9: Radiotap 模板分流
- **控制帧** → 使用最低 MCS 速率（6Mbps）
  - 为什么？确保全网覆盖，即使远距离节点也能接收
- **数据帧** → 使用高阶 MCS（如 MCS 8，144Mbps）
  - 为什么？最大化吞吐量
- **发射时零延迟切换**，提前准备两套模板

---

## Slide 16: 三连发击穿与保护间隔【服务器】

### 关键技术 10: 三连发击穿机制
- **问题**：无线环境可能丢包，控制帧丢失会怎样？
- **方案**：每个控制帧连续发送 3 次
- **接收端**：通过序列号去重
- **效果**：大幅提高控制帧可靠性

### 关键技术 11: 节点切换保护间隔
- **问题**：服务器连续发送两个 Token，硬件队列可能有残留包，导致空中撞车
- **方案**：服务器在发送下一个 Token 前插入 **5ms 保护间隔**
- **为什么是 5ms？** 确保硬件队列完全排空，覆盖传播延迟和系统抖动

<!-- 图表占位：节点切换时序图，标注保护间隔 -->

---

## Slide 17: 客户端接收 Token → 发射控制【客户端】

### 关键技术 12: 零轮询唤醒机制
- 客户端发射线程使用 `std::condition_variable` 等待 Token
- 收到 Token 且目标是自己时，立即唤醒发射线程
- 无 Token 时持续休眠，不占用 CPU
- **效果**：微秒级响应，CPU 零空转

### 流程
1. 客户端收到 Token，解析 target_node 和 window_size
2. 如果是自己 → 在窗口内发送数据
3. 如果不是自己 → 保持静默，等待自己的 Token

---

## Slide 18: 接收端处理 → 状态反馈与窗口调整【客户端】

### 流程
- 发送完毕后，服务器根据实际发送量调整窗口（三态状态机）

### 关键技术 13: 空闲巡逻模式
- **问题**：如果所有节点都静默（队列空），如何发现新数据？
- **方案**：探询间隔 100ms，不是连续探询
- **效果**：10 节点轮询一遍 1150ms，频谱占用 < 15%

<!-- 图表占位：接收端状态机，空闲巡逻时序图 -->

## Slide 19: Token Passing 流程回顾

### 完整流程
```
客户端数据流 → 等待 Token
      ↓
服务器 Token 流 → Token 生成 → 跳过 FEC → Radiotap 分流 → 三连发 → 保护间隔 → 空口
      ↓
客户端发射 → 零轮询唤醒 → 在窗口内发送数据 → 空口
      ↓
服务器确认 → 接收数据 → 更新状态 → 发送下一个 Token
```

# 🛠️ 方案1：底层传输架构工程避坑与优化指南
**—— 针对联邦学习无线空口调度的深度分析与实战建议**

本手册是针对《方案1.md》在工程实现阶段的**核心补丁**。开发者在编码前必须详细阅读以下“设计雷区”及其优化对策，以确保系统在弱信号、高负载环境下的稳定性。

---

## 🚩 雷区 1：“深水池”失效导致的数据静默丢弃
### 【现象描述】
原方案建议通过 `ifconfig wfb0 txqueuelen 50000` 建立 75MB 的内核缓冲区。但在 `wfb-ng` 的原始架构中，`wfb_tun`（读取进程）与 `wfb_tx`（发射进程）通过 **UDP Socket (127.0.0.1)** 通信。
当 `wfb_tx` 逻辑阀门关闭（等待 Token）时，`wfb_tun` 仍会疯狂抽干 TUN 队列并塞向 UDP。一旦 UDP 接收缓冲区（`rmem`）打满，内核会直接丢弃后续包（`ENOBUFS`）。**结果是数据根本憋不住，直接在内部通信环路丢失。**

### 【优化对策】
*   **架构整合（单进程化）**：禁止使用两个独立进程通过 UDP 通信。必须将 `wfb_tun` 的读取逻辑与 `wfb_tx` 的发射逻辑**合并到同一个进程中**。
*   **内存反压机制**：在进程内部建立一个带上限的线程安全队列（如 `std::queue`，上限设为 100~500 个包）。
    *   **逻辑**：当内部队列满时，`tun_reader` 线程必须停止调用 `read()`，从而让压力真正回传到操作系统的 TUN 队列（`txqueuelen`）中。

## 🚩 雷区 2：USB 总线与用户态调度的“相位漂移”
### 【现象描述】
方案定义的 10ms-15ms 微型时间片在工程上极度危险。
1.  **USB 延迟**：普通 USB 无线网卡（RTL8812AU 等）在总线上存在 2~5ms 的调度延迟。
2.  **内核调度**：非实时（Non-RT）Linux 内核的进程上下文切换可能导致数毫秒的抖动。
3.  **追尾风险**：如果 Server 10ms 没收到反馈就切下一个 Token，而 Client 的数据包还在 USB 驱动里打转，就会导致空中射频严重碰撞。

### 【优化对策】
*   **放宽窗口**：将探询令牌和 NACK 分配窗口适度放宽到 **20ms ~ 30ms**。
*   **令牌序号化 (Sequence)**：Token 帧必须带自增序号或时间戳。Client 若收到“过期”或“迟到”的 Token，必须拒绝发射。
*   **保护间隔 (Guard Interval)**：在两个节点的 Token 切换之间，Server 必须强制预留至少 5ms 的全频段静默期。

## 🚩 雷区 3：Linux Kernel 内核态内存溢出 (OOM)
### 【现象描述】
`txqueuelen 50000` 意味着内核要维护 5 万个 `sk_buff` 结构。这不仅消耗 75MB 的载荷内存，还会消耗大量的内核元数据（不可 Swap）。在树莓派等内存有限（1GB/2GB）且正在运行耗能极大的 FL 训练进程的边缘节点上，极易触发 **OOM Killer** 杀掉关键进程。

### 【优化对策】
*   **用户态 Proxy 缓冲**：在 UFTP 与 `wfb0` 之间架设一个轻量级 Proxy 程序。
    *   Proxy 接收来自 UFTP 的极速流量并存储在 **用户态内存**（更安全、受限更小）。
    *   根据底层 `wfb_tx` 反馈的“允许发射”信号，平滑地将数据注入 `wfb0`。
*   **保留限速**：不要完全依赖深水池。在启动 UFTP 时，仍建议设置一个远高于物理带宽但低于内存上限的逻辑限速（例如 50Mbps）。

## 🚩 雷区 4：控制帧（Token）在干扰下的鲁棒性
### 【现象描述】
由于 WiFi Monitor 模式没有硬件级重传，如果 `[TOKEN]` 帧在空中丢失，目标节点将全程静默，Server 则会浪费整个 480ms 的时间片“死等”。

### 【优化对策】
*   **三倍冗余发送**：由于 Token 帧极小（仅几个字节），Server 发送时应**连续喷射 3 遍**。
*   **单包独立降维 (Radiotap MCS 0)**：所有控制帧必须锁定在最保守的调制编码阶数（MCS 0）。**注意：即使 `wfb_tx` 通过命令行参数指定了全局的中/高阶 MCS（用于发数据），在发射 Token 帧时，也必须在 C/C++ 代码中强行绕过该全局配置，为 Token 帧单独拼装一份写死为 MCS 0 的 Radiotap Header。** 重点区别：网卡驱动不支持单包改 TX Power，但**完美支持单包动态改 MCS 速率**，这是 Monitor 包注入模式的标准特性。
*   **固定甜点发射功率 (24dBm)**：切勿尝试在代码中通过 Radiotap 动态修改单包功率（RTL8812AU 等驱动大概率会在底层静默抹杀该配置）。应直接通过系统命令（如 `iwconfig wlan0 txpower 24`）设置全局 24dBm（约 250mW）。在此“甜点功率”下，数据帧的高阶调制波形失真在可容忍范围内，而控制帧配合 MCS 0 已具备足够凶悍的穿透与抗干扰能力，同时避开了网卡过热与 USB 供电断崖的硬件雷区，极大降低开发成本。
*   **Early-Abort（提前侦听切断）**：Server 发出 480ms 令牌后，监听前 50ms。若完全无载荷，判定 Token 丢失，立即下发下一个节点的 Token 以“强行覆盖”。

## 🚩 雷区 5：应用层（UFTP）的状态机超时与 Bufferbloat（缓冲膨胀）
### 【现象描述】
原设想中存在严重的逻辑谬误：“认为 10 个节点排队轮询就会让数据憋上 10 分钟”。事实上，如果我们设定 1 秒的轮询宏观周期，每个节点最长只需要等待约 0.9 秒。
真正导致“数据被憋十几分钟”并引发 UFTP 状态机崩溃断连的罪魁祸首是 **Bufferbloat（缓冲膨胀）**：在旧版无脑的“深水池”设计中（如 50000 的队列长度），当 UFTP 以 10Mbps 的全速疯狂注水，而底层分摊到的物理泄洪率仅为 1Mbps 时，巨大的速度差会导致海量数据全部卡在系统内存里。上万个包带来的超长排队积压时间（长达十几分钟），会彻底击穿 UFTP 的 RTT（往返时间）测算和超时机制，导致其误判网络掉线并主动挂断。

### 【优化对策】
该问题在最新的架构中，已通过**“浅缓冲区 + 快速背压”**机制从根本上得到解决：
*   **架构级根治（丢包触发背压）**：使用 3~4 个时隙的浅水池（`internal_queue_limit: 300~500`）配合极小的内核队列 `txqueuelen 100`。水位满了后不再无脑积压，而是直接产生溢出丢包。这既把最大排队延迟“硬锁死”在了 2~4 秒之内，又利用丢包动作瞬时倒逼应用层 UFTP 退避减速。
*   **精准的微超发限速**：在 UFTP 启动时使用 `-R` 限速，速率锁定为实际可用物理分摊带宽的 1.1~1.2 倍（如 10 节点均分 10Mbps 时设为 1.2Mbps），斩断了导致缓存无限膨胀的源头。
*   **协议容忍度适配**：物理排队延迟已被锁死在极短的安全范围内（不超过 4 秒），我们只需微调 UFTP 的相近控制参数（例如放宽 `GRTT` 平滑上限，调大 `Max Retries` 重传次数），使其包容这几秒钟的轮询抖动即可，彻底告别盲目设置“超长心跳等待”的旧思路。

---
**💡 开发结论：**
本项目的核心难点不在于“发包”，而在于**“憋包”**。开发者应优先实现单进程架构下的**反压流控系统**，这是整个联邦学习系统能否在无线环境下运行的基石。
